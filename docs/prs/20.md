# PR notes: #20 Write the out-of-place conversion script

Branch: `20-write-the-out-of-place-conversion-script`
Plan: `docs/plans/zensical-port-plan.md` (Phases 3 and 4, "Delivery model")

## Cycle 1 (RED): test surface

### Shape of the change

The script is `tools/convert.py`. Public surface:

- `SUBPROJECTS`: the 14 sub-project names, hardcoded, in root-nav order.
- `load_yaml(path)`: parses a mkdocs.yml. The monorepo quotes its
  `!include` values, so they load as plain strings; no custom tag
  handling is required.
- `prefix_nav(nav, prefix)`: recursively prefixes nav page paths,
  leaving titles and external URLs untouched.
- `merge_configs(root_config, sub_configs)`: one merged config dict.
- `copy_content(source, output, subprojects)`: the out-of-place copy.
- `write_zensical_toml(config, path)` / `write_mkdocs_yml(config, path)`.
- `convert(source, output, subprojects=SUBPROJECTS)`: orchestrator; the
  `__main__` entry calls it with the hardcoded roots.

The output is this repository's `zensical/` directory: a self-contained
generated project (`zensical.toml`, transient `mkdocs.yml`, `docs/`)
that the script owns wholesale and regenerates from source on every run.
Nothing outside `zensical/` is written, so the repository's own
documents under `docs/` can never collide with generated content.

Tests: `tools/test_convert.py` (27 tests), driven against a miniature
monorepo fixture (`tools/fixture_tree.py`, built per-test in `tmp_path`)
whose structures are copied from the real corpus: quoted `!include` nav
entries, a conflicting sub-project `extra:` value, duplicate
`pymdownx.highlight` blocks, per-sub-project `mathjax.js` copies, nested
nav sections, a binary asset. Two sub-projects stand in for the fourteen;
the fourteen-name default constant has its own test.

### Decisions encoded in the tests (review these)

1. Root-wins merging. Sub-project configs conflict with the root
   (compiler-user-guide sets `version_maj: 20`, the root 21). The
   monorepo plugin reads only the root's `extra:`/extensions when
   building, so today's rendered site shows 21 on compiler pages;
   root-wins preserves the baseline the Phase 3 diff is judged against.
2. The caption plugin splits by serialiser: retained in the `mkdocs.yml`
   oracle (so the Phase 3 diff keeps the baseline's table numbering),
   excluded from `zensical.toml` (per the #14 decision it is replaced by
   a Python-Markdown extension in Phase 5, #27).
3. Roots are hardcoded, not CLI arguments, per the plan; but the
   `/workspace/` paths from the plan text no longer exist. The script
   derives them from its own location: output is `zensical/` inside the
   repo containing `tools/`, source is the repo's sibling checkout named
   `documentation`. Functions take explicit paths so the tests can drive
   fixture trees.
4. Scope of copying is wider than the issue's one-line summary: the root
   project's own docs (the About pages, `javascripts/mathjax.js`) and
   `documentation-assets/` must be carried too, or the merged config's
   theme/CSS references dangle. The plan's Critical files section implies
   this; the tests make it explicit.
5. Re-run semantics: the script owns the `zensical/` output directory
   wholesale and regenerates it from source, so orphaned and stray files
   do not survive a re-run. Nothing outside `zensical/` is written.

### Open questions (both resolved with the author before GREEN)

1. The site-urls one-link rewrite (#15). Resolved: no script involvement.
   Reading the installed plugin (mkdocs-site-urls 0.3.1) showed its
   default trigger prefix is `site:`, which nothing in the corpus uses,
   so the plugin is a no-op in every current build and dropping it needs
   no content change. The one root-absolute link predates the migration
   and is broken on the production monorepo site; the relative rewrite is
   to be upstreamed to the source repo as a one-line content fix before
   Phase 3 (tracked on #15). The script copies verbatim; Principle 2
   holds without exception.
2. Generated content versus this repository's own documents. Resolved:
   the output is the self-contained `zensical/` directory (decision taken
   with the author), so the repository's `docs/` is never part of a
   build. Encoded in the surface by the stray-file re-run test and the
   copy tests' layout.

### Cycle 1 revision (after review)

The review at `docs/reviews/20.md` returned Request changes with two
Major findings and one Minor; the layout decision taken with the author
(the self-contained `zensical/` output directory) landed between the
review snapshot and this revision. Changes:

1. extra_css folds root-only, not superset (review Major 1). The
   reviewer measured, and this revision re-verified, that the corpus's
   only sub-only extra_css entries are shared-code-files-user-guide's
   three `style/*.css` lines, whose targets do not exist in its docs;
   the monorepo build ignores sub extra_css, so a superset fold would
   add three dangling stylesheet links to every page and break the
   Phase 3 zero-churn gate. This deviates from the plan's "folded into
   the superset" wording deliberately: the zero-churn gate outranks it.
   The fixture now carries a dangling sub-only entry and
   takes_extra_css_from_the_root_alone pins the fold.
2. Ownership boundary (review Major 2). Superseded by the `zensical/`
   layout: the script owns the output directory wholesale and writes
   nowhere else. Pinned from both sides by
   rerun_removes_stray_files_planted_in_the_output and the new
   writes_nothing_outside_the_output_directory.
3. The missing-sub-project contract is pinned to FileNotFoundError
   naming the missing sub-project (review Minor).

RED re-run after the revision: 29 tests, 0 passed (19 failures, 10
fixture-setup errors), all via NotImplementedError from the stubs or

## Cycle 2 (GREEN): implementation

### What was implemented

All functions in `tools/convert.py` per the tested contract. Notable
internals: `!include` entries are matched by a strict regex
(`^!include \./<name>/mkdocs\.yml$`) and replaced by the sub-project's
prefixed nav; the plugin exclusions are two frozensets (`monorepo` and
`site-urls` dropped from the merged config, `caption` additionally from
`zensical.toml` only); `copy_content` removes the output directory and
rebuilds it, which is what makes re-runs deterministic and orphan-free;
TOML serialisation uses `tomli-w` (pinned in
`tools/requirements-dev.txt` as a script runtime dependency).

### Surface amendment during GREEN (author's direction)

The `emit_mkdocs_yml` flag and its test were removed as dead code: the
script itself is retired at cutover, so no production run can ever pass
False. Both serialisations are always written. A None-stripping helper
in the TOML path was removed on the same reasoning before it landed; the
real-corpus run confirmed the corpus produces no None values. The suite
is 28 tests after the amendment.
