# PR notes: #30 CI pipeline for the Zensical build and versioned deploy

Branch: `30-ci-pipeline-versioned-deploy`
Plan: `docs/plans/zensical-port-plan.md` (Phase 7). Part of #6.

## Nature of this issue

This is infrastructure/config, not unit-testable code: a GitHub Actions
workflow plus a build-requirements file. There is no red/green test surface.
Verification is a `workflow_dispatch` staging run, which is an outward action
held for explicit approval (see "Not done here"). The transformation steps that
could carry a bug were verified locally.

## What it does

`.github/workflows/publish.yml` (manual `workflow_dispatch`) builds the
committed `zensical/` project with Zensical and deploys a versioned layout to
`gh-pages` via the squidfunk/mike fork. Steps: install pinned deps, determine
version, substitute copyright variables, optionally override `site_url` for a
staging host, gate on `check_flatten`, then `mike deploy`.

Inputs: `version_override` (else `zensical.toml` `extra.version_majmin`),
`set_as_latest`, and `site_url_override` (for staging/test hosts). Inputs are
passed to steps via `env:`, not interpolated into scripts, to avoid workflow
script injection.

## Dependency gap closed

The build loads Python-Markdown extensions (`pymdownx.*`,
`markdown_tables_extended`, `dyalog_caption`) and Zensical itself, which were
only pinned in the source monorepo. The monorepo is archived at cutover, so the
output repo now owns them: `tools/requirements-build.txt` pins
`zensical==0.0.48`, `pymdown-extensions==10.21.3`,
`markdown-tables-extended==0.1.3`; the caption extension installs via
`pip install -e tools/`; the mike fork stays in `tools/requirements-deploy.txt`.

## Deviations from the issue text

- `documentation-assets` is a submodule at `zensical/docs/documentation-assets`
  (the corporate style; #41), so the checkout uses `submodules: recursive` to
  fetch it before the build reads its CSS/fonts. (An earlier draft wrongly
  claimed no submodule step; the submodule handling is required.)
- The copyright `envsubst` is retargeted at `zensical.toml` (was `mkdocs.yml`);
  the committed config carries the `$CURRENT_YEAR`/`$BUILD_DATE`/`$GIT_INFO`
  placeholders and CI substitutes them, exactly as the source workflow did.
- No `mkdocs.yml` fallback exists to retire; CI reads `zensical.toml` directly.
- Added: a `check_flatten` gate step (Phase 7 calls for gating on it) and a
  `site_url_override` input so staging/test hosts get their own canonical
  (deferred from #28).

## Verification (local, where possible)

- YAML parses; the step list is as intended.
- Copyright `envsubst` on the real `zensical.toml`: substitutes the three
  variables, leaves valid TOML, and leaves `site_url` intact.
- `site_url_override` regex on the real config: replaces
  `https://docs.dyalog.com/` with the staging URL, still valid TOML.
- The mike-fork versioned deploy itself was proven end to end on a throwaway
  two-version project in #28 (correct `/<ver>/`, `versions.json`, alias, root
  redirect).
- Dependency closure: a fresh venv with only `tools/requirements-build.txt`
  plus the caption extension (`pip install -e tools/`) builds the full corpus
  (3,034 pages, the 20-issue baseline), with the submodule checked out. No
  missing transitive dependency, so CI does not silently rely on the source
  monorepo's stack.
- `check_flatten` is clean on the current tree.

## Not done here (outward actions, held for approval)

- Merging the workflow to `main` (a `workflow_dispatch` workflow must be on the
  default branch before it can be dispatched).
- Enabling GitHub Pages on `xpqz/zensical-dyalog` (source: `gh-pages`) so the
  staging URL serves. The workflow pushes the `gh-pages` branch regardless;
  Pages only affects serving.
- Running the staging `workflow_dispatch` (with `site_url_override` set to the
  xpqz Pages URL) to confirm `/21.0/` resolves and the selector switches. This
  is the issue's done-when and the safe on-xpqz test discussed earlier.

## Review outcome

`docs/reviews/30.md`: Approve with minor changes. Applied:

- Dropped the "for now" narrative marker from the workflow header comment.
- Rewrote `requirements-deploy.txt`'s header, which still pointed at the source
  monorepo for the build stack; it now points at `requirements-build.txt` (this
  PR is what moved that ownership into the output repo).
- Hardened the `site_url` override: the `re.sub` replacement is now a function,
  so backslashes/quotes in the URL are never treated as regex group references
  (a pathological URL now fails loudly as invalid TOML rather than corrupting).

Notes left as accepted: two full Zensical builds per run (gate then deploy)
serve different purposes and the mike fork does not expose the checker stream to
merge them; the version validation is per-line but the input is a trusted
`workflow_dispatch` value; actions are pinned to major tags (conventional).
