# PR notes: #28 Wire zensical build into the versioned deploy

Branch: `28-wire-zensical-build-into-versioned-deploy`
Plan: `docs/plans/zensical-port-plan.md` (Phase 6). Part of #5.

## Approach (verified before writing code)

The plan assumed mike could not build Zensical. That is true of upstream
jimporter/mike (2.2.0 builds mkdocs/properdocs), but there is a dedicated
fork, `squidfunk/mike`, that swaps the build step to `zensical build`. It is
the approach Zensical's own docs prescribe, chosen here (pinned to a commit)
over a hand-rolled deploy script.

Verified end to end with the fork (pinned commit
`2d4ad799442f4592db8ad53b179bfb33db8c69ac`) on a throwaway two-version deploy:

- `mike deploy <ver>` runs `zensical build --clean` with `MIKE_DOCS_VERSION`
  set and reads `zensical.toml`. It builds our project natively; no mkdocs.
- gh-pages layout is correct: `/<ver>/` directories, `versions.json` (mike
  schema), the `latest` alias, and a root `index.html` redirecting to the
  default.
- Zensical version-prefixes `site_url`: the 21.0 page's canonical came out
  `https://docs.example.com/21.0/`. This only happens when `site_url` is set.
- The version selector is native: the page's `__config` carries the
  `version` provider and references `versions.json`. No theme override needed
  for the selector (the outdated-version warning banner is #29).

Two facts this establishes for the committable work:

1. `zensical.toml` must carry `site_url`; the fork does not set it. The source
   config has none. The production canonical host is `docs.dyalog.com`
   (`tools/jenkins/RELEASE.md`: production is `docs.dyalog.com/<ver>/`, staging
   is `dyalog.github.io/documentation/<ver>/`). Internal links are relative, so
   the same `site_url` is correct for both contexts; the canonical points at
   production, as it should.
2. The deploy environment must have `zensical` on PATH (the fork calls it as a
   bare command), i.e. the venv activated or its bin on PATH.
