# PR notes: #36 Rewrite raw-HTML headings to Markdown headings

Branch: `36-rewrite-raw-html-headings-to-markdown`
Plan: `docs/plans/zensical-port-plan.md` (Phase 5). Resolves the #18 finding; unblocks #23.

## Problem

~3,020 raw-HTML page-title headings (`<h1 class="heading">` with inner styling
spans, via md_in_html) across 3,009 pages leave those pages without an ATX
heading. Zensical then derives the `<title>` and search-entry title from the
prettified filename rather than the heading text (the #18 divergence). Upstream
is not an option, so the conversion fixes it.

## Approach (verified in a spike, Zensical 0.0.48)

`convert.py` rewrites each raw `<h1 ...>INNER</h1>` to a Markdown ATX heading,
carrying the attributes via attr_list and escaping Markdown-special characters
in INNER:

```
<h1 class="heading"><span class="name">Button</span> <span class="right">Object</span></h1>
->
# <span class="name">Button</span> <span class="right">Object</span> {: .heading}
```

Confirmed by the spike: Zensical then uses the heading text as `<title>` and
search title; the rendered `<h1 class="heading">` and its inner spans survive
(styling intact, plus a heading anchor); and escaping the backslash keeps APL
command text such as `R←f\[K]Y` verbatim (the naive rewrite mangles it to
`R←f[K]Y` because the ATX inline parser eats `\[`).

Corpus shapes covered: `class="heading"` with span `name` (1689), `name`+`right`
(700), `command`+`name` (619); tail of `class="example"` (7), bare `<h1>` (3),
`id`+`class` (1), `align=center` (1).

This deliberately relaxes the conversion script's copy-verbatim invariant
(Principle 2) for one deterministic, tested format conversion (raw-HTML heading
-> Markdown heading), which is what a MkDocs->Zensical port is for. The source
tree is still never written; the rewrite is applied to the copied `.md` files
in the output.

## Cycle 2 (GREEN): implementation

### What was implemented

`rewrite_h1` scans the markdown line by line, tracking fenced code blocks, and
rewrites each standalone raw `<h1 ...>...</h1>` line (outside fences) to a
Markdown ATX heading. The heading element is parsed with BeautifulSoup
(`html.parser` backend), not by regex: `h1.attrs` gives the attributes
structurally (mapped to attr_list: `id`->`#id`, `class`->`.name`, other->
`key="value"`), and every text node in the subtree (including inside the
styling spans) is escaped for Markdown specials via `NavigableString`
replacement, so APL command text survives the heading's inline parser. The
inner spans are re-serialised verbatim, so the rendered `<h1 class="heading">`
with its `name`/`right`/`command` spans is structurally identical to the
original: the language-reference-guide header CSS still binds. `copy_content`
applies the rewrite to the copied `.md` files (source untouched; files without
a raw `<h1>` are left byte-for-byte as copied). `beautifulsoup4` is declared in
`requirements-dev.txt` as a script runtime dependency.
