# PR notes: #29 Version-selector theme override

Branch: `29-version-selector-theme-override`
Plan: `docs/plans/zensical-port-plan.md` (Phase 6). Part of #5.

## Scope

The version **selector** renders natively in Zensical's theme (confirmed in
#28: `base.html` emits the `mike` version config and the bundle draws the
dropdown from `versions.json`). The one thing that is not native is the
**outdated-version warning banner** on non-latest versions. This issue provides
that as a theme override, which is what the plan anticipated.

## Mechanism

`base.html` already renders a warning aside, but only when the `outdated` block
is non-empty:

```
<div data-md-component="outdated" hidden>
  {% if self.outdated() %}
    <aside class="md-banner md-banner--warning">
      ... {% block outdated %}{% endblock %} ...
      {% include "partials/javascripts/outdated.html" %}
```

`outdated.html` unhides the banner when the bundle's version JS has flagged the
current version as outdated (it compares against `versions.json` and sets
`__outdated` in sessionStorage). So the override only needs to fill the block.
The authoritative Zensical snippet:

```html
{% extends "base.html" %}

{% block outdated %}
  You're not viewing the latest version.
  <a href="{{ '../' ~ base_url }}">
    <strong>Click here to go to latest.</strong>
  </a>
{% endblock %}
```

`custom_dir = "overrides"` points Zensical at it. The link goes up one level to
the site root, which redirects to the default version, so it does not depend on
the `latest` alias.

Location: a top-level `overrides/` (sibling to `docs/`), not the `docs/overrides`
the Zensical docs show in their example. Verified in a spike: a template under
`docs/` is published as a static file in the built site (an unused raw template
leaking to `/overrides/main.html`), whereas a top-level `overrides/` builds the
same banner without leaking. This is also the conventional Material layout. The
banner only renders when `config.extra.version` is set (the enclosing condition
in `base.html`), which the generated config already provides
(`extra.version.provider = "mike"`).

## Delivery

`convert.py` owns `zensical/` wholesale, so it must produce the override and set
the config:
- copy a committed template (`tools/overrides/main.html`) to
  `zensical/overrides/main.html`, and
- set `theme.custom_dir = "overrides"` on the merged config.

The override is a project artefact kept under `tools/`, not sourced from the
monorepo (which never had a wired override; its
`documentation-assets/overrides/main.html` overrides `site_meta` only and is not
referenced by any `custom_dir`).

  `zensical.toml` and the new `overrides/main.html`; content is untouched.
