# PR notes: #21 Generate the flattened project and validate under Zensical

Branch: `21-generate-the-flattened-project-and-validate-under-mkdocs`
Plan: `docs/plans/zensical-port-plan.md` (Phases 3 and 4)
Closes: #21 and #22 (merged, see below)

## Scope change agreed before this cycle

Two decisions taken with the author reshape the issue from its original text:

1. Validation is Zensical-only. The obsolete source-repo checkers
   (`tools/utils/check_links.py`, `dangling_links.py`) are not used, and no
   MkDocs baseline build or HTML diff is produced. Faithfulness is judged by
   Zensical's own built-in link/anchor checker. The MkDocs build-time
   comparison is dropped: Zensical is already accepted as fast enough. Note
   this leaves site-root-absolute links unchecked (Zensical does not validate
   them); the one known such link is tracked separately on #15 and a
   follow-up issue covers guarding against new ones after cutover.
2. Forward-looking, no fallback. Tools read the MkDocs monorepo and write
   Zensical; they carry no backwards-compatibility path. The conversion
   script therefore stops emitting the transient `mkdocs.yml` oracle, which
   existed only as a MkDocs fallback and diff oracle. `zensical.toml` is the
   sole generated config.

Consequently #21 and #22 are delivered together on this branch: generate the
flattened tree, prove the flatten adds no new checker issues under Zensical,
and commit the tree plus `zensical.toml` to the repo. The separate #22 build
step collapses into the same validation.

## Cycle 1 (RED): test surface

### Shape of the change

Two pieces of work, each with a failing surface:

1. `tools/convert.py` emits only `zensical.toml`. `write_mkdocs_yml` and its
   call are removed; `caption` is dropped from the merged config alongside
   `monorepo` and `site-urls` (it has no Zensical module and is replaced by a
   Python-Markdown extension in Phase 5, #27), so the `zensical.toml`-only
   plugin exclusion no longer needs a separate oracle carve-out.
2. `tools/check_flatten.py`, a Zensical-native validation tool. It builds the
   project, normalises each checker diagnostic to `<type> @ <file>:<line>:<col>`,
   and diffs that set against the committed baseline `known-issues.txt`. Any
   issue outside the baseline fails the check. Its pure functions
   (`parse_issues`, `load_baseline`, `diff_issues`) are unit-tested against
   captured checker output; `run_build`/`main` shell out to `zensical build`
   and are exercised as the integration gate in Cycle 2, not unit-tested.

`known-issues.txt` is the baseline of 28 pre-existing issues, enumerated from
an actual `zensical build` of the current flattened tree: 8 caption
`_table-N` anchors (restored by the Phase 5 caption numbering extension, #27)
and 20 broken relative links/anchors already present in the source. The
site-root-absolute link tracked in #15 is deliberately not here: Zensical
does not validate absolute URLs, so its checker never reports it. That link
remains in the source and is not touched by this branch; #15 stays open on
its own track.

### Cycle 1 revision (after review)

The tests-only review (`docs/reviews/21.md`) returned Request changes with
three Major findings and two Minors. Changes:

1. Baseline provenance corrected (Major 1). The 28 entries are 8 caption
   anchors plus 20 pre-existing broken links/anchors. The earlier claim of a
   site-root-absolute link "fixed upstream" was wrong on both counts:
   Zensical does not validate absolute URLs so the checker never reports that
   link, and it is unchanged in the source. The `known-issues.txt` header and
   the baseline paragraph above are rewritten; issue IDs are removed from the
   data-file header; #15 stays on its own track.
2. The silent-pass modes are now in the tested contract (Major 2). A new
   `parse_issue_count` reads the checker's "N issues found" summary, and a new
   pure `assess(output, returncode, baseline)` fails the gate on a non-zero
   build exit, a missing summary line, a parsed-count/summary disagreement, or
   any issue outside the baseline. `run_build` now returns
   `(output, returncode)`; `main` takes an injectable `runner` so its exit
   policy is unit-tested (clean → 0, new issue → 1) without a real build. The
   severity regex also matches Error, not only Warning.
3. The scope change is recorded on issues #21 and #22 as a done-when comment
   before this branch closes them (Major 3).
4. `known-issues.txt` header states the shrink-only pruning policy: resolved
   entries are removed so a later regression reads as new (Minor 2).
5. The root-absolute-link blind spot (Minor 1) is deferred to a follow-up
   issue rather than expanded into this gate: it is a different guarantee
   (scan copied markdown for `](/` and `href="/`) with its own baseline, and
   it matters most after cutover when this repo becomes the authoring home.

RED re-run after the revision: 47 tests, 27 passed, 20 failed (18
`check_flatten` via `NotImplementedError` from the stubs, 2 `convert`

## Cycle 2 (GREEN): implementation

### What was implemented

`copy_content` now excludes `.git` when copying: `documentation-assets` is a
submodule in the source, so its `.git` gitlink would otherwise be dragged into
the content tree. Pinned by `test_does_not_carry_vcs_metadata_into_the_output`.

`check_flatten.py`: `parse_issues` (ANSI-stripped, severity line paired with
the `╭─[ loc ]` header), `parse_issue_count` (the "N issues found" summary),
`load_baseline`, `diff_issues`, and the pure `assess`. `run_build` invokes
Zensical through the running interpreter (`python -m zensical`) so it does not
depend on PATH.

`convert.py`: `write_mkdocs_yml` removed; `caption` added to the dropped
plugins in `merge_configs`; `write_zensical_toml` simplified (the exclusion is
now applied once, upstream); module docstring updated. `convert` writes only
`zensical.toml`.

The plan is updated: Principles 1, 3 and 4, the "Validation oracle" section
(was "Reuse existing tooling"), the Delivery model, Phases 3-4 (merged), 7 and
8, the Critical files list and the Verification strategy no longer describe the
`mkdocs.yml` oracle, the MkDocs baseline diff or the legacy checkers.
