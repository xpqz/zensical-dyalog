# Port Dyalog documentation from MkDocs monorepo to Zensical

## Context

`/workspace/documentation/` is the source for Dyalog's user documentation: a MkDocs
site of ~3,050 markdown files across 14 sub-projects, stitched together by the
`mkdocs-monorepo-plugin`. The monorepo split exists for one reason only: full MkDocs
builds were too slow for authors to iterate on such a large corpus.

Zensical (from the Material for MkDocs team) has differential builds that remove that
constraint, so the monorepo is no longer needed. Zensical does not support the monorepo
plugin (its "Subprojects" feature is on the roadmap but not built), so the port must
flatten the 14 sub-projects into one Zensical project. Zensical reads `mkdocs.yml`
natively and uses the same Python-Markdown parser as MkDocs, so most markdown extensions
carry over; the risk is concentrated in a handful of plugins and content conventions.

Intended outcome: one flattened Zensical project (the new single-stack authoring home) that
builds the whole corpus, renders every feature the current site relies on, and stays
demoable at every commit. The MkDocs monorepo remains a read-only source during the
migration; the flattened project is generated out-of-place into a separate location and
becomes the authoring home at cutover.

### Decisions taken with the user
- Initialise/vendor the `documentation-assets` submodule so styling can be verified.
- First demoable milestone is a single-sub-project spike, not the whole corpus.
- Defer versioning (mike / `/21.0/` URLs); produce an unversioned build for early demos.
- Adopt `zensical.toml` as the sole build config, generated by the conversion script. No
  `mkdocs.yml` is emitted (revised after the go/no-go passed: the tooling is Zensical-only,
  forward-looking, with no MkDocs fallback).
- Validate the flatten with Zensical's own built-in link/anchor checker, not a MkDocs baseline
  diff or the legacy source-repo checkers (revised alongside the config decision above).
- The conversion is out-of-place: the script reads the monorepo as a read-only source and
  writes the flattened project into a separate output directory. The source repo is never
  mutated (no in-place `git mv`).
- CHM and PDF outputs are being deprecated and are out of scope; the plan does not port
  `chm/mkdocs2chm.py` or `pdf/mkdocs2pdf.py`.
- The full MkDocs baseline build is not run: Zensical is accepted as fast enough (go/no-go
  passed, issue #11), so the ~60-minute MkDocs cold-build number is no longer a comparison the
  plan depends on.

### Key facts verified during planning
- The monorepo mounts each sub-project at a URL path equal to its directory name
  (e.g. `object-reference/docs/objects/button.md` is served at `/object-reference/objects/button/`).
  886 cross-sub-project links are authored against this geometry, e.g.
  `](../../../language-reference-guide/system-functions/dq)`. Therefore copying
  `<sub>/docs/*` to `docs/<sub>/*` in the output project preserves every public URL and
  every relative link. This is the pivot that makes a zero-churn flatten possible.
- No `overrides/`, no `custom_dir`, no MkDocs `hooks:`, no Jinja control blocks. `macros`
  is used only for simple `extra:` variable substitution (`{{version_majmin}}` etc.).
- `site-urls` is enabled but only one site-root-absolute link exists in content
  (`](/implementation-details/the-webconfig-file/)`), so the plugin can be dropped after
  fixing that single link.
- Highest residual risks: the non-standard `markdown_tables_extended` (colspan via empty
  cells, rowspan via italic-whitespace cells) registering under Zensical; the MathJax
  re-typeset hook bound to Material's `document$` observable; and versioning (mike wraps
  `mkdocs build`).

## Guiding principles (apply to every phase)
1. Validate the flatten with Zensical's own checker. Zensical reports every unresolved
   internal link and anchor during a build; the flatten is faithful when it introduces no
   issue outside a committed baseline of issues already present in the source. No MkDocs
   baseline build or HTML diff is produced.
2. The script copies files from source to the output tree, never edits them. Preserve the
   directory-name-equals-mount-point invariant so URLs and `{#id}` anchors stay byte-identical.
3. `zensical.toml` is the sole build config, generated by the conversion script into the
   output project and committed there. No `mkdocs.yml` is emitted: the tooling reads MkDocs
   config and writes Zensical config, with no MkDocs fallback or diff oracle.
4. Forward-looking only. The tooling carries no backwards-compatibility path and depends on
   no MkDocs build at run time. The conversion script is retained until cutover only because
   it regenerates the output tree from an updated source, then retired.

## Go/No-Go gate (the decisive criterion)

The entire migration exists to fix author build times, so the go/no-go decision is a
build-performance criterion, not a rendering one: **does the flattened structure build fast
enough on the full corpus to be usable by authors?** Concretely:

- Author edit loop (primary gate): after editing one page, Zensical's incremental rebuild
  under `zensical serve` completes fast enough for edit-save-preview. Target: low
  single-digit seconds. This is the number the monorepo was invented to work around, so it
  is the number that must improve.
- Cold full build (secondary gate): a from-scratch build of all ~3,050 files completes in a
  time acceptable for CI. The MkDocs full cold build is taken as ~60 min (the pain the
  monorepo was invented to work around); Zensical should improve materially on this.

This gate is front-loaded via a build-speed probe (Phase 1a) run on the full flattened tree
before investing in feature parity. If the incremental rebuild is not materially faster than
MkDocs on the same flattened tree, the premise fails and we stop or reconsider before doing
the feature work. Feature-parity phases (2, 4, 5) only proceed once this gate is passed.

Result (measured, Zensical 0.0.48, full 3,030-page flattened tree, 32 GB machine): cold
build 30.6s (39.2s wall), peak 6.9 GB max RSS (macOS reports a 17.5 GB peak memory
footprint including compressed pages); warm no-change batch rebuild 14.2s; rebuild after a
one-page edit under `zensical serve` 8.7s with default validation (the built-in link/anchor
checker dominates and its cost scales with corpus size) and 0.6-0.8s with
`validation: false`. Verdict: GO. The author edit loop is sub-second with validation off
during authoring, against MkDocs rebuilding the whole site (~60 min) on every edit.
Consequence for Phases 4 and 7: validation off under `zensical serve`, validation on in
batch/CI builds, where the 30s cold build absorbs it. Full builds need at least 16 GB RAM.
Bonus: Zensical ships built-in link/anchor checking, which is the validation oracle for the
flatten (28 issues on the probe tree: 8 caption anchors restored in Phase 5, and 20 broken
links pre-existing in the source; Zensical does not validate site-root-absolute links, so
the one such link is neither reported nor counted here). Full-corpus numbers supersede the
528-page subset measurement taken in the 7.7 GB sandbox and its extrapolations; details on
issue #11.

## Validation oracle
Zensical's built-in link/anchor checker is the regression oracle for the flatten. It runs as
part of every build and reports each unresolved internal link and anchor. `tools/check_flatten.py`
wraps a `zensical build`, normalises the reported issues, and fails when any appears outside a
committed baseline of pre-existing source issues (`known-issues.txt`). The source repo's legacy
QA scripts (`check_links.py`, `dangling_links.py` and the rest under `tools/utils/`) are not
used: they belong to the MkDocs stack being retired.

## Delivery model: the out-of-place conversion script
The flatten and config conversion are performed by a single Python script, not by hand, and
run out-of-place: it reads the monorepo checkout (read-only, a sibling directory named
`documentation/`) and writes the flattened project into the self-contained `zensical/`
directory of the output repo [xpqz/zensical-dyalog](https://github.com/xpqz/zensical-dyalog)
(default branch `main`). The script owns `zensical/` wholesale and regenerates it on every
run; it writes nowhere else, so the output repo's own documents under `docs/` can never
collide with generated content. The source repo is never mutated. The script hardcodes the
Dyalog-specific assumptions (the 14 sub-project names and their config shape) to stay minimal:
no command-line arguments or configuration. `<output>` below denotes the generated project
directory `zensical/` inside the output repo. The script:
- copies `<sub>/docs/*` from source into `<output>/docs/<sub>/*`,
- consolidates the duplicated `mathjax.js` into one canonical `<output>/docs/javascripts/mathjax.js`,
- merges the 14 sub-project configs plus the root into one merged config, and
- serialises that merged config to `<output>/zensical.toml`, the sole build config. No
  `mkdocs.yml` is emitted.

It is idempotent and re-runnable against a clean checkout, so regenerating the output tree from
an updated source is deterministic. It is retained until cutover so the output tree can be
regenerated from an updated source; it is retired once Zensical is production.

Config-format decision: `zensical.toml` is the authoritative and only committed build config in
the output project (per Principle 3). CI reads it directly; there is no `mkdocs.yml` consumer to
keep working.

## Phases

Each phase ends in a demoable, committable state. Mapped to the repo's red/green workflow,
the "test" for structural phases is a `zensical build` whose built-in checker reports no
issue outside the committed baseline (the "Validation oracle" above); the "green" is that
build passing with zero new broken links or anchors.

### Phase 0 - Baseline, assets, tooling
Goal: make the current state reproducible and diffable, and unblock styling verification.
- `git submodule update --init --recursive` to check out `documentation-assets` (all CSS,
  APL385 font, custom-admonition styling, logos, favicon live there).
- Create a pinned venv from `tools/requirements-docs.txt`; add Zensical (pin the exact
  version) alongside, without removing the MkDocs stack.
- A full `mkdocs build` golden baseline was scoped here originally but not built: validation
  is Zensical's own checker (the "Validation oracle" above), not a MkDocs HTML diff, so no
  reference `site/` is needed. The MkDocs cold-build time (~60 min) was only ever used as the
  go/no-go comparison in Phase 1a, which has passed.
Demo: current site builds and renders with correct branding; `zensical --version` runs in
the same venv.
Commit: tooling only (add Zensical to requirements, document venv setup). Source repo
otherwise untouched.

Phases 1 and 1a are independent and may run in parallel. Phase 1 is informational (can
Zensical render the extension stack at all); Phase 1a is the actual go/no-go gate (is the
flattened structure fast enough). Neither blocks the other: a rendering failure in the spike
does not stop the speed probe, and a passing speed probe does not require the spike to have
succeeded. Only the Phase 1a gate governs whether feature work proceeds.

### Phase 1 - Zensical spike on one small sub-project
Goal: prove Zensical ingests this project's config + extension stack and renders with the
real theme/CSS, cheaply.
- Use `release-notes` (7 files): small but exercises macros vars, caption, admonitions,
  `md_in_html`, MathJax config.
- Build it standalone with Zensical from its existing `release-notes/mkdocs.yml`, wiring in
  the real `documentation-assets` CSS/font.
Demo: a stakeholder clicks through 7 styled pages that look like production.
Commit: none required (local spike); optionally a short notes entry. Value is confirming
Zensical can ingest and render this project's stack at all (a prerequisite for the speed
probe, not itself the go/no-go).

### Phase 1a - Build-speed probe (the go/no-go gate)
Goal: measure whether the flattened structure builds fast enough to be usable, on the full
corpus, before investing in feature parity.
- Use the conversion script (or a quick disposable variant) to generate a full flattened tree
  out-of-place. Rendering imperfections are acceptable here; the goal is a build that
  completes, not a correct one. Suppress or stub any extension that hard-fails the build so a
  timing can be taken.
- Measure the two gate numbers: cold full `zensical build` time, and the incremental rebuild
  time under `zensical serve` after touching one page. Compare against `mkdocs build` /
  `mkdocs serve` on the same generated tree as the baseline.
Demo/verify: a short table of cold-build and incremental-rebuild times for Zensical vs MkDocs
on the full flattened corpus.
Go/No-Go: proceed only if the incremental rebuild meets the author-edit-loop target (low
single-digit seconds) and is materially faster than flattened MkDocs. If it does not, stop
and reconsider before any feature work.
Commit: none (generated tree is disposable); record the timings in the notes.

### Phase 2 - Risky-feature verification matrix
Goal: turn every unknown into a verdict (works / workaround / must-write-a-module), on the
spike plus a few hand-picked worst-case pages.

| Feature | Verify / decide | Expected verdict and fallback |
|---|---|---|
| `markdown_tables_extended` (colspan/rowspan) | Build a known colspan+rowspan page; confirm the extension registers under Zensical's Python-Markdown (by entry point or dotted module path) and emits merged `<td colspan/rowspan>`. | Top risk. If it will not register: a thin markdown-extension shim or a preprocessor. |
| macros variables | Confirm Zensical's committed `macros` module does `extra:`-based substitution (pure lookups, no Python calls). | Works. Ensure version vars exist in the merged config. |
| `caption` table numbering | No Zensical module. Prefer a CSS-counter solution (`counter-increment` + `caption::before`). Decisive check: grep content for numbered cross-references to tables (e.g. "see Table 5", "Table {index}", "in Table 3"). If none exist, CSS-counter suffices; if any exist, CSS counters cannot back a cross-reference and a module/preprocessor that assigns stable numbers is required. | CSS-counter (engine-agnostic) if no numbered cross-refs; otherwise a numbering module. Defer implementation to Phase 5. |
| `site-urls` | Convert the single absolute link to relative and drop the plugin. | Retired trivially. |
| Custom admonitions (ucmdhelp/windows/unix/macos/linux/Legacy) | Confirm Zensical emits the same `admonition <type>` classes so submodule CSS matches. | Low risk; verify class names. |
| APL / nonAPL / other fences | `use_pygments:false, pygments_lang_class:true` emits `class="language-apl"`, highlighted client-side. Confirm Zensical emits identical classes. | Low risk if class emission matches. |
| Raw-HTML `<h1 class="heading">` titles (3,000+ pages) | Verify nav labels (explicit nav), `<title>` tag and search index get sensible titles when the H1 is HTML via `md_in_html`. | Medium risk; verify at object-reference scale in Phase 5. |
| MathJax + `navigation.instant` hook | `docs/javascripts/mathjax.js` subscribes to Material's `document$`. Verify Zensical's instant-nav exposes an equivalent. | Top risk. Fallback: Zensical post-navigation hook, or drop instant nav on math sections, or per-page MathJax load. |

Commit: optional notes entry recording verdicts. The `site-urls` link fix is applied to the
generated output (the source stays read-only), or upstreamed to source as a one-line content
fix if preferred.

**Verdicts** (evidence on the linked issues; measured with Zensical 0.0.48 against MkDocs
1.6.1 on identical test projects):

- `markdown_tables_extended`: works; emits merged colspan/rowspan cells identical to
  MkDocs. No shim needed (#12).
- macros variables: works; `extra:`-based substitution confirmed. The merged-config vars
  are checked when the merged config exists (#13, #20, #21).
- `caption`: a numbering module is required, not CSS counters, twice over: eight
  `[Table N](#_table-N)` cross-references exist (one page), and without the plugin the
  `Table:` line renders as a plain paragraph, so no `<caption>` element exists for CSS to
  number. A small Python-Markdown extension reproduces the caption element, per-page
  numbering and `_table-N` anchors in Phase 5 (#14, #27).
- `site-urls`: exactly one absolute link confirmed corpus-wide; its target is in the same
  sub-project, so a one-line relative rewrite retires the plugin from all 13 configs
  (#15, #20).
- Custom admonitions: works; identical lowercased class emission, assets CSS binds (#16).
- APL/nonAPL/other fences: works; identical `language-*` class emission (#17).
- Raw-HTML `<h1 class="heading">` titles: diverges. Zensical falls back to the prettified
  filename for `<title>` and search-entry titles where MkDocs uses the explicit nav label;
  nav labels and body indexing match. Resolution (upstream fix, front-matter injection, or
  acceptance) tracked on #18; gates #23.
- MathJax + `navigation.instant`: works; Zensical exposes `window.document$` and the
  existing hook needs no change (#19; runtime re-typeset check stays in #25).

### Phase 3-4 - Generate the flattened project and build it under Zensical
Goal: produce one flattened project with a unified nav from the monorepo and build the whole
site under Zensical from the generated `zensical.toml`, proving zero new link/anchor breakage.
Phases 3 and 4 are delivered together: validation is Zensical's own checker, so there is no
separate MkDocs-validation step between generating the tree and building it under Zensical.

Mechanism: run the conversion script (see "Delivery model"). Reading the 14 hardcoded
sub-projects from source, it writes into the output directory:
- `<output>/docs/<sub>/*`, copied from `<sub>/docs/*`. Content files are copied, never edited,
  so links and `{#id}` anchors are untouched.
- `<output>/docs/javascripts/mathjax.js`, one canonical copy; the per-sub-project duplicates
  are simply not carried over. The merged config's `extra_javascript` references this single
  path. This is an asset consolidation, not a content-link edit, so it does not breach the
  "files copied, never edited" invariant for content.
- `<output>/zensical.toml`, the merged config: each `"!include ./<sub>/mkdocs.yml"` replaced by
  that sub-project's nav path-prefixed with `<sub>/`, preserving the 6 top-level headings and
  titles; each sub-project's `markdown_extensions` and `extra:` folded into the superset with
  the root value winning conflicts (collapsing the duplicate `pymdownx.highlight` block noted
  in several sub-configs); `extra_css` taken from the root alone, because the monorepo build
  ignores sub `extra_css` and the corpus's only sub-only entries dangle in source, so folding
  them in would change the rendered baseline; the `monorepo`, `site-urls` and `caption` plugins
  removed (`caption` is replaced by a Python-Markdown extension in Phase 5). The remaining
  plugins `search`, `privacy`, `macros`, `minify` map to committed Zensical modules.

Demo/verify: `zensical build` the generated `<output>` tree; `tools/check_flatten.py` confirms
its built-in checker reports no issue outside `known-issues.txt` (the pre-existing source
issues), and fails on a non-zero build exit or a parsed-count/summary disagreement.
Build environment: the full-corpus build needs at least 16 GB RAM (measured in Phase 1a:
6.9 GB max RSS, 17.5 GB peak memory footprint on macOS); 7.7 GB containers OOM. Size the CI
runner accordingly, and run CI/batch builds with validation on, `zensical serve` with
validation off (see the go/no-go result).
Commit: the generated `zensical/` directory (tree plus `zensical.toml`) in the output repo.
The source monorepo is untouched. The conversion script and `tools/check_flatten.py` are
committed under the output repo's tooling (retained until cutover).

### Phase 5 - Retire scale-and-rare risks
Goal: confirm the features that only bite at scale or in edge cases.
- object-reference (1,695 pages): verify `<title>`, search indexing and nav labels across a
  large sample; confirm any `search: exclude` front matter is honoured.
- APL/nonAPL/other fences bind the client-side highlighter site-wide.
- Implement the MathJax instant-nav fallback if `document$` is unavailable; verify
  equations re-typeset on client navigation.
- All custom admonition types render with correct CSS.
- Apply the chosen `caption` CSS-counter (or numbering module) solution so tables show
  "Table N:" again.
Demo/verify: guided click-through of object-reference, a math page under instant nav, an
APL code page, each custom admonition, and a numbered table.
Commit: one per feature fix for clean review (in the output repo).

### Phase 6 - Versioning and URL-structure preservation
Goal: restore `/21.0/`-prefixed publishing and the version selector (deferred from early
phases per the agreed plan).
- The published layout is per-version directories on `gh-pages` + `versions.json`, served
  by nginx clean URLs. mike wraps `mkdocs build`, so it cannot build the Zensical site
  directly.
- Replace mike's build step with `zensical build` into a versioned output dir, then reuse
  mike (or a small script) for the `gh-pages` version-dir + `versions.json` + default
  alias layout. If Zensical's theme does not render a selector reading `versions.json`,
  keep the selector as a small theme override until Zensical ships versioning. "Small theme
  override" here means a single MiniJinja template partial (a header fragment) that fetches
  `versions.json` and renders the version dropdown, dropped into the theme's override
  directory, not a custom module or a forked theme.
- Leave the nginx clean-URL config unchanged (URLs preserved by Phase 3 geometry).
Demo/verify: deploy two versions to a staging `gh-pages`; confirm `/21.0/` URLs resolve and
the selector switches versions.

### Phase 7 - CI pipeline
Goal: publish the Zensical build from CI, keeping the release pipeline green.
- Build a CI pipeline for the output repo that builds `<output>` with `zensical build` and
  deploys via the Phase-6 versioned layout.
- Carry over the `documentation-assets` submodule handling and the copyright substitution
  (the `$BUILD_DATE`/`$GIT_INFO`/`$CURRENT_YEAR` `envsubst` step), retargeting it at the
  `zensical.toml` copyright field. Update or replace `test-jenkins-simulation.yml`.
- Gate the pipeline on `tools/check_flatten.py`: the build fails CI if the checker reports any
  issue outside `known-issues.txt`.
Demo/verify: a `workflow_dispatch` staging run publishes a Zensical build of the output repo.

### Phase 8 - Cutover and decommission
Goal: make the flattened Zensical project the production authoring home and retire scaffolding.
- Flip production deploy to Zensical building the output repo; authors switch to editing the
  output project.
- Drop MkDocs and the removed plugins (`monorepo`, `site-urls`, `caption`) from the output's
  requirements; retire the conversion script.
- Archive the monorepo source repo.
Demo/verify: production build+deploy from Zensical only; `check_flatten` clean.

## Verification strategy (end to end)
0. Go/No-Go gate (Phase 1a): cold-build and incremental-rebuild timings on the full flattened
   corpus; proceed only if the author edit loop is fast enough. Passed (issue #11).
1. Structural phases (3-4): `zensical build` the generated tree, with `tools/check_flatten.py`
   confirming the built-in checker reports no issue outside `known-issues.txt` and the build
   exits cleanly.
2. Feature phases: the Phase-2 torture-test page plus guided click-throughs
   (object-reference at scale, a math page under instant nav, APL fences, each custom
   admonition, a numbered table).
3. CI/versioning phases: `workflow_dispatch` staging run publishing a Zensical build; two-version
   staging deploy to confirm `/21.0/` URLs and the version selector.

## Top residual risks (highest first)
1. `markdown_tables_extended` (colspan/rowspan) may not register under Zensical - affects
   many reference tables. Verify in Phase 2; fallback is a shim/preprocessor.
2. MathJax + `navigation.instant` hook depends on Material's `document$` observable, which
   Zensical's instant-nav runtime may not expose. Verify in Phase 2, fix in Phase 5.
3. Versioning gap - Zensical versioning is roadmap-only; mike is coupled to `mkdocs build`.
   Needs a build+deploy bridge and possibly a custom version selector.
4. Raw-HTML `<h1 class="heading">` titles on 3,000+ pages - `<title>`/search/nav behaviour
   under Zensical must match MkDocs. Verify at object-reference scale.
5. `caption` numbering loss - CSS-counter workaround is cosmetic and will not support
   numbered cross-references if any exist. Verify none do via the Phase 2 grep for numbered
   table references; if any exist, escalate to a numbering module/preprocessor.
