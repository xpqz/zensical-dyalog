# Review: Issue #36 - Rewrite raw-HTML headings to Markdown headings (RED)

## Summary

Tests-only (RED) review of the `rewrite_h1` transform surface. The tests are
correct specifications: every assertion is satisfiable by the design's intended
GREEN output, verified by rendering the intended transform results through the
same extension pipeline the tests use (`attr_list`, `md_in_html`,
`pymdownx.highlight`, `toc`). The RED run reproduces the documented 9 failed /
59 passed, ruff is clean, and the new files comply with the project writing
conventions. The stub correctly stays unwired so the existing copy tests remain
green while the new integration test fails on un-rewritten output. Findings are
coverage gaps to fold into GREEN, chief among them a fenced-code test that a
broken implementation could still satisfy. None block committing the RED tests.

## Findings

### Major: The fenced-code test does not prove rewriting still happens outside the fence

`test_does_not_touch_raw_h1_inside_a_fenced_code_block` feeds an input whose
only raw `<h1>` sits inside a fenced block; the page title (`# Real Title`) is
already ATX. The single assertion checks the in-fence `<h1>` survives verbatim.
Because there is no raw `<h1>` outside the fence, a GREEN implementation that
bails out entirely on encountering any fence (returns the whole file unchanged)
would pass this test while silently failing every real corpus page that has
both a raw-HTML page title and an HTML code example. Object-reference pages
routinely have both, so this is exactly the case the transform must get right.

Recommendation: use an input that contains a genuine raw `<h1 ...>` title
outside the fence and a fenced raw `<h1>` example, then assert both that the
title became an ATX heading (rewritten) and that the fenced one is byte-for-byte
preserved. That pins the skip to the fenced region rather than the whole file.

Location: tools/test_rewrite_h1.py:81-88

### Minor: The general attribute mapping path (non-class, non-id) is untested

The contract maps `class` to `.x`, `id` to `#id`, and any other attribute to
`key=value` in the attr_list block. The tests exercise only the class and id
branches (NAME_RIGHT, COMMAND, and the id+class test). The corpus `align="center"`
shape (1 page) is the only occurrence that drives the general-attribute branch,
and no test covers it, so GREEN will implement that branch untested.

Recommendation: add a case for an `<h1>` carrying a non-class, non-id attribute
and assert the rendered heading keeps it.

Location: tools/test_rewrite_h1.py (test set)

### Minor: A file with more than one raw <h1> is untested, though the contract says "each"

The corpus holds roughly 3,020 raw `<h1>` across 3,009 pages, so about eleven
pages carry more than one. The docstring and PR contract say the transform
rewrites "each raw `<h1 ...>INNER</h1>`", but no test distinguishes global
substitution from first-match-only. A regex applied without a global flag would
pass every current test yet leave the second heading raw on those pages.

Recommendation: add an input with two raw `<h1>` headings and assert both are
rewritten.

Location: tools/test_rewrite_h1.py (test set)

### Note: The most common shape (class="heading", single name span, 1689) is not tested in isolation

It is structurally covered: NAME_RIGHT exercises the class plus multi-span case,
and the id+class test uses a single `name` span. The code path is exercised, so
no separate test is required.

### Note: The class="example" shape (7 occurrences) is not tested separately

It travels the same class-mapping path as `class="heading"`. No action needed.

### Note: test_leaves_markdown_without_a_raw_heading_unchanged already passes in RED

Because the rewrite is not yet wired into `copy_content`, this test passes now
via the verbatim copy rather than by failing first. It is a GREEN regression
guard against the rewrite corrupting plain Markdown, and the byte-preservation
tests (`test_copies_each_subproject_docs_tree_to_docs_sub_preserving_bytes`,
`test_two_runs_produce_byte_identical_output`) reinforce the same invariant. The
PR doc's 9-failed count already accounts for this being green. No action.

### Note: APL heading text is literal, not a computation

`R←f\[K]Y` is a corpus command signature (Scan with an axis), not an expression
to evaluate, so there is no expected APL value to verify with dyalog-script. The
property that matters, verbatim survival through the Markdown inline parser, is
verified by rendering: the backslash before `[` is preserved rather than eaten
by the inline escape, which is the whole point of the escaping step.

## Verification

- Tests: RED reproduced. `python -m pytest tools/ -q` gives 9 failed, 59 passed.
  The 8 `test_rewrite_h1` failures are `NotImplementedError` from the stub; the
  one `test_convert` failure is the `AssertionError` in
  `test_rewrites_raw_html_headings_in_output_markdown` (copied page still holds a
  raw `<h1>` because the rewrite is not wired). Matches docs/prs/36.md.
- Lint: `ruff check tools/` reports all checks passed.
- Satisfiability: rendered the design's intended transform outputs through the
  test's extension set. All unit assertions hold: `<h1 class="heading">` with
  both inner spans is preserved, the ATX heading gains `id="button-object"`, the
  APL backslash survives (`R←f\[K]Y`), id+class map correctly, and the bare
  `<h1>` renders. A raw (un-rewritten) `<h1>` gains no toc id, confirming the
  title test discriminates a real ATX heading from a raw passthrough.

## Recommendation

Approve with minor changes. The tests are correct and satisfiable specifications
of the intended behaviour. Fold the coverage additions into the GREEN cycle,
prioritising the strengthened fenced-code test since it is the one gap a broken
implementation could pass.

# Review: Issue #36 - Rewrite raw-HTML headings to Markdown headings (GREEN)

## Summary

Implementation (GREEN) review of commit e3e4f41 on top of RED commit 8a453e2.
The change is correct, complete, and verified end-to-end. `rewrite_h1` scans the
markdown line by line, tracks fenced code blocks, and rewrites each standalone
raw `<h1>...</h1>` outside a fence into `# INNER {: attrs}`, parsing the heading
with BeautifulSoup (not regex): attributes map structurally to attr_list and
every text node is escaped for Markdown specials so the styling spans survive
and APL command signatures pass the inline parser intact. The three coverage
gaps from the RED review (the Major fenced-code test plus the two Minor
attribute/multiple-heading tests) are all addressed, and the RED review's core
worry, an implementation that bailed on any fence, is refuted: the-page-load-event.md
keeps its fenced `<h1>` via fence tracking while its title (if any) would still
convert. The full corpus build confirms the outcome: 3,016 headings rewritten,
the four code/example `<h1>` correctly preserved, titles and search entries now
derived from the heading text, styling intact, and check_flatten clean against
the 20-entry baseline. Findings are one Minor coverage gap and three Notes; none
block merge.

## Findings

### Minor: The standalone heuristic that protects httprequest's embedded <h1> is pinned only end-to-end

`_is_standalone_h1` returns True only when the stripped line both starts with
`<h1` and ends with `</h1>`. That `startswith` clause is what keeps
httprequest.md's two embedded raw `<h1>` raw: `<body><h1>Simple Form</h1>` ends
with `</h1>` but starts with `<body>`, and the APL string literal
`[8]    response,←'<h1 align="center">Hello '` ends with neither. Both are left
alone correctly, but only check_flatten (the full corpus build) exercises this;
no fast unit test feeds a non-standalone line that ends in `</h1>`.
`test_leaves_content_without_a_raw_h1_untouched` covers the general
no-standalone-heading case but not the `</h1>`-terminated embedded shape. A
regression that relaxed the condition to test only `endswith("</h1>")` would
keep the whole unit suite green while mangling httprequest's HTML sample.

Recommendation: add a unit test feeding a line such as
`<body><h1>Simple Form</h1>` (and optionally the APL-string-literal shape) and
assert it is returned unchanged.

Location: tools/convert.py:226-231; tools/test_rewrite_h1.py (test set)

### Note: Fence tracking is line-oriented and does not model indented code blocks

`_FENCE_RE` recognises opening and closing markers of three or more backticks or
tildes, and the loop toggles on the marker's first character. It does not model
4-space indented code blocks, nor CommonMark's rule that a closing fence be at
least as long as its opener. This is adequate and verified for the current
corpus (exactly the four expected raw `<h1>` survive, one pair via this fence
tracking in the-page-load-event.md and code-behind.md), but a future standalone
raw `<h1>` inside an indented code block would be rewritten. No action for this
corpus.

Location: tools/convert.py:87-88, 212-223

### Note: The Markdown-special escape set is scoped to the corpus and active extensions

`_MD_SPECIAL_RE` escapes ``\ ` * _ [ ] { }``, the inline specials that bite in
this corpus and the extension set (`attr_list`, `md_in_html`,
`pymdownx.highlight`, `toc`). It is not exhaustive of every pymdownx inline
(for example `~` for tilde/strikethrough or `^` for caret), but no corpus
heading triggers those and the full build renders clean, including the APL
backslash case. No action.

Location: tools/convert.py:89-90, 249-250

### Note: The _is_standalone_h1 docstring undercounts the surviving raw headings

The docstring says "The two corpus `<h1>` occurrences that sit inside code ...
are not standalone, so they are left alone." That is precise for the two this
function handles (httprequest), but the corpus leaves four raw `<h1>` in total;
the other two are protected by fence tracking, not by this predicate. A reader
scanning for how many survive could misread it. Optional wording tweak.

Location: tools/convert.py:227-229

regress silently. The Notes are optional polish and need no change for this
corpus.
