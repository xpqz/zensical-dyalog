# Review: Issue #27 - Caption numbering extension (tests-only RED)

## Summary

This is the tests-only RED review. The change adds a stub Python-Markdown
extension (`tools/dyalog_caption.py`, whose `CaptionTreeprocessor.run` raises
`NotImplementedError`), a seven-test behavioural surface
(`tools/test_dyalog_caption.py`), and two wiring tests in
`tools/test_convert.py`. The tests faithfully encode the mkdocs-caption 1.3.0
contract captured in `docs/prs/27.md`: a `Table: <text>` paragraph before a
table becomes `<table id="_table-N">` with a `<caption style="caption-side:top">Table N: <text></caption>`
first child, numbered per document from 1, and authored `[Table N](#_table-N)`
cross-references resolve to the table id. The suite fails for the right reasons
(the seven extension tests via `NotImplementedError`, the two convert tests via
`AssertionError` because the extension is not yet injected), with no import or
collection errors, matching the documented RED expectation. The assertions were
sanity-checked against the real parser: `markdown_tables_extended` emits a bare
`<table>` with no attributes, so the exact-string `<table id="_table-1">`
assertions are achievable by adding a single `id` attribute rather than
depending on attribute ordering. The tests are well-formed and ready to commit.
Findings are minor and can be folded into the GREEN cycle.

## Findings

### Minor: No test pins per-page numbering reset across separate renders

The contract is per-page numbering from 1.
`test_numbering_increments_per_document` verifies the counter increments within
one document (`_table-1`, `_table-2`), but no test verifies that a second,
independent render restarts at `_table-1`. A GREEN implementation that stored
the counter at module or class level (rather than per treeprocessor instance),
or a build that reused one Python-Markdown instance across pages, would leak
numbering into the next page: its first table would render `_table-9...` and
its `[Table 1](#_table-1)` cross-reference would silently break. Because the
affected corpus has all eight tables on one page, `check_flatten` would not
catch this regression. Since resolving cross-references is the entire point of
this work, add a test that two separate `render()` calls each produce
`<table id="_table-1">`.

Location: tools/test_dyalog_caption.py:38 (test_numbering_increments_per_document)

### Note: `_CAPTION_RE` is DOTALL with a greedy capture, and is untested at RED

`_CAPTION_RE = re.compile(r"^Table:\s*(.+)$", re.DOTALL)` ships in the stub but
is exercised by no test (the treeprocessor raises before using it). With
`re.DOTALL` the `.+` is greedy across newlines, so a multi-line paragraph node
whose text begins `Table:` would capture the trailing lines into the caption
text. Corpus captions are single-line, so this is unlikely to bite, but the
matching behaviour is unverified. Confirm the intended matching (single line vs
whole paragraph) is covered by a test when `run` is implemented in GREEN.

Location: tools/dyalog_caption.py:28

### Note: test docstring references the issue-numbered PR doc

The module docstring points the reader at `docs/prs/27.md` for the contract.
Per the project comment-hygiene convention, source and test comments should not
lean on issue/PR-numbered references. This is a soft reference (a file path, not
a `#27` mention) and the docstring already names the contract ("the
mkdocs-caption contract"), so this is low priority, but consider stating the
contract inline or referencing the captured contract without the numeric id so
the test reads standalone.

Location: tools/test_dyalog_caption.py:6

### Note: settle the build-time import decision in favour of an installable package

`docs/prs/27.md` leaves open how the Zensical build resolves the extension by
name. The unit tests import `dyalog_caption` directly (pytest inserts `tools/`
on the path) and are unaffected. Endorse the author's lean to option 1: a
minimal installable package listed in the build requirements, importable
uniformly across the venv, `check_flatten`, and CI, with no env coupling. Note
that the string used in `markdown_extensions` must equal the importable module
name (`dyalog_caption`); the two convert tests correctly pin that exact string,
so the packaging choice must keep the module importable under that name.

Location: docs/prs/27.md (Open decision); tools/test_convert.py:221, tools/test_convert.py:299

### Note: assertion tightness is appropriate, one loose regex is acceptable

The exact-equality caption-text assertion (`caption.group(1) == "Table 1:
Commonly used Variables"`) and the exact `<table id="_table-1">` string are
correctly tight given the base extension emits a bare `<table>`. The
caption-side check (`r'<caption[^>]*caption-side:top[^>]*>'`) is looser than the
documented `style="caption-side:top"` and would pass on any attribute carrying
that substring, but paired with the other caption tests this is adequate and
follows the substring style already used elsewhere in `test_convert.py`. No
change required.

Location: tools/test_dyalog_caption.py:35

## Verification

  48 passed: seven `tools/test_dyalog_caption.py` failures via
  `NotImplementedError` from the stub, two `tools/test_convert.py` failures via

- Lint: `ruff check tools/` passes (all checks passed).
- Contract sanity check: `markdown_tables_extended` renders a bare `<table>`

## Recommendation

Approve with minor changes.

The tests are correct, well-scoped, and fail for the right reasons. Before or
during the GREEN cycle, add the per-page numbering-reset test (Minor); the
remaining items are Notes. Proceed to commit the tests and settle the build-time
import as option 1.

# Review: Issue #27 - Caption numbering extension (implementation GREEN)

## Summary

This is the implementation review, over commit 30d8e30 on top of the RED
commit d930ab3. The change greens the RED surface: `CaptionTreeprocessor.run`
pairs each `Table: <text>` paragraph with the table element immediately
following it, sets `id="_table-N"` on the table (N counted per document from 1),
inserts `<caption style="caption-side:top">Table N: <text></caption>` as the
table's first child, and removes the paragraph. `merge_configs` appends the
`dyalog_caption` extension to the merged `markdown_extensions`, and a minimal
`tools/pyproject.toml` packages the module so Python-Markdown imports it by name
at build time regardless of cwd. The regenerated `zensical.toml` gains the one
extension line; `known-issues.txt` shrinks 28 to 20 as the eight `_table-N`
anchors now resolve. All three RED findings are addressed: the numbering-reset
test is present and passing (`test_numbering_restarts_at_one_for_each_render`),
the `DOTALL` matching is settled (the regex matches one paragraph's text and the
`" ".join(...split())` fold normalises internal whitespace), and the docstring
no longer references the issue-numbered PR doc. The build-time import landed as
option 1 (installable package). The full suite passes (58), lint is clean, the
integration gate exits 0, and the rendered page carries all eight captions,
table ids, and resolving cross-references. The implementation is correct and
well-scoped. The findings below are all Notes requiring no change.

## Findings

### Note: Captions containing inline Markdown would lose their marked-up text

`run` inspects only `child.text` and matches `^Table:\s*(.+)$` against it. If a
caption paragraph contained inline markup (for example `Table: The *foo*
variable`), Python-Markdown would place `Table: The ` in the paragraph's `.text`
and the emphasised run in a child `<em>` element with a tail. The regex would
capture only `The`, emit `Table N: The`, silently drop the emphasised portion,
and still consume the paragraph and number the table. All eight corpus captions
are plain text (verified against the source), and the extension is explicitly
scoped to reproduce this one corpus behaviour, so this does not bite today. It
is recorded because mkdocs-caption itself supported richer captions: if future
corpus content adds an inline-formatted caption, its text would be truncated
without warning. No change required for this issue.

Location: tools/dyalog_caption.py:52-54, tools/dyalog_caption.py:64

### Note: requirements-dev.txt documents the editable install but does not perform it

The build imports the extension by module name, so `dyalog_caption` must be
installed (option 1, as chosen). `tools/requirements-dev.txt` records the step
only as a comment (`pip install -e tools/`); it is not an installable line, so
`pip install -r tools/requirements-dev.txt` alone does not make the extension
importable. There is no CI workflow in the repository yet, so nothing is broken
now, and the install is documented in three places (pyproject.toml,
requirements-dev.txt, the PR doc). When CI or build automation lands, it must
run `pip install -e tools/` explicitly, or add `-e ./tools` as a real
requirement line. No change required for this issue.

Location: tools/requirements-dev.txt:12-14

### Note: treeprocessor priority 5 coincides with the toc treeprocessor

`register(CaptionTreeprocessor(md), "dyalog_caption", 5)` uses priority 5, the
same priority the toc extension registers its treeprocessor at. The two operate
on disjoint elements (toc scans headers; this scans `Table:` paragraphs and
their following tables), so relative ordering does not affect correctness, and
the full corpus build confirms this. Recorded so the shared priority reads as a
known coincidence rather than an accident. No change required.

Location: tools/dyalog_caption.py:80-81

### Note: tree is mutated during root.iter(), which is safe here

`run` removes the caption paragraph and inserts the caption element while
iterating `root.iter()`. This is safe: the inner loop iterates a snapshot
(`children = list(parent)`), and each parent's removals all complete before the
generator resumes and reads that parent's live child list, so no table is
skipped and no element is processed twice. The inserted `<caption>` is later
yielded by the outer iterator but skipped (tag is not `p`). The corpus build,
which nests tables inside deep section structures, exercises and confirms this.
No change required.

Location: tools/dyalog_caption.py:45-67
