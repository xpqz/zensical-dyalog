# Review: Issue #30 - CI pipeline for the Zensical build and versioned deploy

## Summary

Implementation review of commit `1b9473e`. This issue is infrastructure/config
(a `workflow_dispatch` GitHub Actions workflow, a build-requirements file, and
runbook updates), so there is no red/green test surface and none is expected;
verification is by local checks plus a staging dispatch held for approval. The
workflow is correct and safe on the points that matter: the version is read

three intended variables, the `site_url` override targets the single canonical
line, the mike invocation and `contents: write` permission are appropriate, and
inputs reach steps through `env:` rather than being interpolated into `run`
scripts, which closes the workflow-injection vector. The load-bearing
transforms were re-verified locally (YAML parses, the `tomllib` path resolves to

one convention slip and small hardening/accuracy notes; none block merge.

## Findings

### Minor: "for now" narrative marker in the workflow header comment

The header comment reads "Manual trigger only for now; staging is
xpqz.github.io/...". Per the project writing conventions, "for now" is a
narrative marker to avoid; the comment should state the invariant ("Manual
trigger only."). Everything else in the comment is fine.

Location: `.github/workflows/publish.yml:4`

### Minor: stale monorepo reference in requirements-deploy.txt

This PR moves ownership of the build stack (Zensical, the Markdown extensions)
into `tools/requirements-build.txt` so the output repo no longer depends on the
archived source monorepo. `tools/requirements-deploy.txt` (not touched by this
commit) still carries a comment saying the build "also needs Zensical (pinned in
the source monorepo's `tools/requirements-docs.txt`)", which now contradicts the
new ownership. Outside this commit's diff, but flagged because it is exactly the
dependency-ownership story this change establishes; a one-line comment fix keeps
the two requirements files consistent.

Location: `tools/requirements-deploy.txt` (header comment, lines 4-6)

### Note: site_url override replacement is not escaped

The override step builds the replacement with an f-string and passes it to
`re.sub(r'^site_url = ".*"$', f'site_url = "{url}"', ...)`. `re.sub` treats
backslashes in the replacement as group references, so a URL containing a
backslash would raise `re.error` or corrupt the value, and a literal `"` would
produce malformed TOML. `site_url_override` is a trusted maintainer input via
`workflow_dispatch` and real URLs contain neither character, so the risk is
low; a `lambda` replacement (or `re.escape` on the injected text) would remove
the sharp edge entirely.

Location: `.github/workflows/publish.yml:98`

### Note: version validation matches any line, not the whole value

`echo "$VERSION" | grep -qE '^[0-9]+\.[0-9]+$'` succeeds if any line of
`$VERSION` matches, so a multi-line `version_override` would pass validation and
carry a newline into `GITHUB_OUTPUT` and then into `mike deploy "$VERSION"`.
The input is maintainer-supplied via `workflow_dispatch`, so this is a
robustness note rather than a real exposure; rejecting embedded newlines (or
matching against the whole string) would tighten it.

Location: `.github/workflows/publish.yml:65`

### Note: two full Zensical builds per run (gate then deploy)

`check_flatten.py` runs `zensical build` to diff diagnostics against the
baseline, and `mike deploy` then runs `zensical build` again to produce the
published artefact, so each dispatch builds the corpus twice. This is
acceptable: the two builds serve different purposes (fail the pipeline on a
link/anchor regression before any push, versus produce-and-push the versioned
site), and the mike fork does not expose the diagnostic stream that would let
one build do both. The cost is build time, not correctness, and the gate
correctly runs against the already-substituted config.

Location: `.github/workflows/publish.yml:107-121`, `tools/check_flatten.py:134-146`

### Note: actions pinned to major tags rather than commit SHAs

`actions/checkout@v4` and `actions/setup-python@v5` are pinned to major-version
tags. This is conventional and fine; SHA-pinning would harden the supply chain
further but is not required by any repo convention. No change needed.

Location: `.github/workflows/publish.yml:38,45`

### Note: injection posture and deploy safety are handled correctly

Recorded as a positive observation. All three `workflow_dispatch` inputs reach
their steps via `env:` and are read as shell/Python variables, never
interpolated into a `run` script; `site_url_override` is only tested in an
Actions expression (`if:`) and read via `os.environ`. `permissions` is the
minimal `contents: write` needed to push `gh-pages`, and the `docs-publish`
concurrency group with `cancel-in-progress: false` serialises deploys so
concurrent version pushes do not race. The copyright substitution, `site_url`
override, and version read all run before both the gate and the deploy, so the
published config is the one that was gated.

Location: `.github/workflows/publish.yml` (general)
