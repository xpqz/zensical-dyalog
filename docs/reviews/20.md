# Review: Issue #20 - Write the out-of-place conversion script

Stage: tests-only (RED), Cycle 1.

## Summary

The 27-test surface is well shaped: every behaviour named in the issue and
the plan's delivery model is pinned by at least one test, the fixture
structures each verify against the real corpus, the fourteen-name constant
matches the root nav's `!include` order exactly, and the recorded RED run
reproduces exactly (17 failures plus 10 errors, 0 passes). Two gaps must be
closed before the tests are committed, and both correspond to questions the
PR doc itself raises: the fold behaviour for sub-only `extra_css` entries
(the real corpus contains exactly one such case, and it is dangling in
source) and the ownership boundary of the generated `docs/` tree (the
current suite passes if the script wipes `docs/` wholesale, which would
delete this repository's own documents). Each needs a decision and a
pinning test.

## Findings

### Major: extra_css folding is unspecified for the one sub-only case the corpus contains

`test_folds_extra_css_superset_preserving_root_order` exercises only
entries that duplicate the root's list, so it does not define what happens
to an entry that exists in a sub-config but not in the root. Measured
against the real corpus, exactly one such case exists:
`shared-code-files-user-guide` declares `style/main.css`,
`style/extra.css` and `style/admonition-ucmdhelp.css`, and no `docs/style/`
directory exists in that sub-project, so all three entries dangle in
source (the monorepo build ignores sub `extra_css`, which is why this goes
unnoticed today). A superset fold imports three dangling stylesheet links
into every page of the flattened site and breaks the Phase 3 zero-churn
gate. Every non-dangling sub entry is already in the root's list, so
root-only folding and superset folding differ only by these three entries.
Recommendation: fold `extra_css` root-only (or drop entries whose source
file does not exist), add the structure to the fixture, and pin it with a
test.

Location: tools/test_convert.py:165, tools/fixture_tree.py

### Major: the ownership boundary of the generated docs/ tree is untested

PR-doc decision 5 states that the script owns the subtrees it generates
under `docs/` and leaves files it does not own untouched, but no test
encodes the second half. An implementation that deletes `docs/` wholesale
and rebuilds it passes every current test, including
`test_rerun_removes_output_files_whose_source_disappeared`, and in
production would delete `docs/plans/` and `docs/prs/`, the repository's own
documents, which live inside the generated `docs_dir` (open question 2).
Resolve the repo-documents question first (moving them out of `docs/`, for
example to `notes/`, is the option that needs no config carve-outs), then
add a test that a pre-existing file the script does not own survives a
re-run.

Location: tools/test_convert.py:76, docs/prs/20.md (decision 5, open question 2)

### Minor: missing-sub-project failure contract is looser than "fails loudly"

`test_raises_when_an_expected_subproject_is_missing` accepts any
`Exception` whose message mentions the missing name. Pin the exception type
the script commits to (`FileNotFoundError` or `ValueError` both fit) so the
contract is defined for callers and future tests.

Location: tools/test_convert.py:84

### Note: on open question 1, upstream the site-urls rewrite

The tests encode verbatim copy, which is the right surface. Upstreaming the
one-line content fix to the source monorepo before Phase 3 keeps the
copy-never-edit invariant intact and needs no documented exception in the
script. Recommend that option.

### Note: root-wins is corroborated by corpus measurement

Across all 14 sub-configs, the only `extra:` conflicts with the root are
the version keys (`version_maj` 20 and `version_majmin` 20.0 in ten
sub-projects), and there are no sub-only markdown extensions, no sub-only
plugins beyond the dropped `site-urls`, and no sub-only `extra_javascript`
entries. Root-wins therefore preserves the rendered baseline exactly as
decision 1 argues, and the superset fold of `markdown_extensions` is a
no-op on the real corpus.

### Note: fixture fidelity and the fourteen-name constant verified

The quoted `!include` form, the conflicting `extra:` value (compiler 20
against root 21), the duplicated `pymdownx.highlight` block (15 configs),
the per-sub-project `mathjax.js` copies (10 sub-projects) and `site-urls`
living in sub-configs only (13 sub-projects, not the root) all match the
real corpus. The default tuple in
`test_default_subproject_list_is_the_fourteen_from_the_root_nav` matches
the root nav's 14 `!include` lines in order.

## Verification

- Tests: fail (expected RED). 27 tests, 0 passed: 17 failures plus 10
  errors, matching the PR doc's recorded distribution (25
  `NotImplementedError`, 10 of which surface as fixture-setup errors
  through the shared `merged` fixture, plus 2 assertion failures). Run in
  the local venv (Python 3.11, pytest 9.1.1).
- APL semantics step: not applicable, the suite contains no APL
  expressions.

## Recommendation

Request changes. Scope: resolve the two decisions above (extra_css fold,
repo-documents location), pin each with a test, and optionally narrow the
missing-sub-project exception type. Everything else is ready to commit.
