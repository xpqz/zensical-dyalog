# Review: Issue #29 - Version-selector theme override (tests-only RED)

## Summary

This is the tests-only RED review. Three tests are added to
`tools/test_convert.py` to define the delivery of the outdated-version warning
banner as a theme override: two config-level assertions that
`theme.custom_dir == "docs/overrides"` (once on the merged config, once on the
serialised `zensical.toml`), and one filesystem assertion that
`docs/overrides/main.html` is produced, extends `base.html`, and overrides the
`outdated` block. The tests fail for the right reasons and in the shape the PR
doc predicts: `python -m pytest tools/ -q` gives 3 failed, 73 passed, with two
`KeyError: 'custom_dir'` (the merged theme has none yet) and one `AssertionError`
(the override file is absent). `ruff check tools/` is clean. The tests assert
observable behaviour (config values and output files), reuse the existing
`merged`, `source_tree`, and `out_dir` fixtures, follow the established
sentence-style naming, and carry invariant-style comments with no issue IDs,
emojis, or em-dashes. The two-level (merged plus serialised) pairing mirrors the
existing `site_url` tests, so it is deliberate conformity, not redundancy. Two
observations follow: one Minor looseness in the override-content assertion, and
one Note on the chosen override location.

## Findings

### Minor: The override-content test does not pin the warning message or the latest-version link

`test_places_the_version_warning_theme_override` asserts only two structural
markers: `'extends "base.html"'` and `"block outdated"`. Both are satisfied by an
empty override such as `{% extends "base.html" %}{% block outdated %}{% endblock %}`.
That is exactly the do-nothing case the feature exists to prevent: `base.html`'s
default `outdated` block is already empty, so an override that overrides the block
with nothing renders an empty banner and the warning never appears. The point of
this override is the content it injects (the "not viewing the latest version"
text and the link to latest), and the test does not guard it. Because the GREEN
artefact is a committed, human-reviewed template, the practical risk today is
low, but the test is a weak regression guard: a later edit that empties the block
would still pass. Consider adding an assertion that the block is non-empty, for
example that the text contains the latest-version link (`base_url`, or an `<a`
anchor), so the test pins the behaviour and not just the template's skeleton.

Location: `tools/test_convert.py:73-79`

### Note: custom_dir places the override directory inside docs/

Tests 1 and 2 pin `custom_dir` to `docs/overrides`, which nests the theme
override directory inside the documentation source tree, and test 3 confirms the
template lands at `docs/overrides/main.html`. In MkDocs-family builds, everything
under the docs directory is treated as documentation content, so there is a risk
the build publishes the override template as a served page (for example at
`/overrides/main.html`). The conventional Material layout keeps overrides in a
sibling directory (`custom_dir: overrides` at the project root), and since
`convert.py` owns the whole output tree wholesale it could equally emit the file
at `output/overrides/main.html` with `custom_dir = "overrides"`. The two test
expectations are internally consistent with each other (the file location matches
the `custom_dir` value), so this is not a defect in the tests as written. It is a
design choice the tests encode: if GREEN verification shows Zensical publishes the
nested template as content, the fix would change these expected values, so the
placement is worth confirming during the GREEN build check.

Location: `tools/test_convert.py:284-287`, `tools/test_convert.py:350-354`,
`tools/test_convert.py:73-79`

### Note: The issue's staging acceptance is outside this unit-test surface

Issue #29 is worded around a version selector and its done-condition is "the
selector switches versions on staging". The PR doc reframes the scope: the
selector is native in Zensical's theme (established in #28), so #29 delivers only
the outdated-version warning banner as an override. That reframing is documented
and reasonable, and the selector's switch-on-staging behaviour is a manual
staging check that cannot be exercised at the `convert.py` unit level. The test
surface correctly targets what is testable here (the override file is produced
and wired via `custom_dir`); the staging acceptance remains a manual verification
step, consistent with the PR doc's Cycle 2 plan. No action.

Location: `docs/prs/29.md`, `tools/test_convert.py`

## Verification

- Tests: fail as expected for the RED stage. `python -m pytest tools/ -q` gives
  3 failed, 73 passed. Failures:
  `test_places_the_version_warning_theme_override` (`AssertionError`, override
  file absent), `test_sets_theme_custom_dir_for_the_version_warning`
  (`KeyError: 'custom_dir'`), `test_zensical_toml_sets_theme_custom_dir`
  (`KeyError: 'custom_dir'`). This matches the PR doc's predicted RED run.
- Lint: `ruff check tools/` passes.

## Recommendation

Approve with minor changes. The tests are correct, fail for the right reasons,
and conform to the repository's conventions. Tightening the override-content
assertion (the Minor finding) would make the third test guard the banner's
actual behaviour rather than its skeleton; the two Notes are for awareness and
GREEN-stage verification and do not block committing the tests.

# Review: Issue #29 - Version-selector theme override (implementation GREEN)

## Summary

The implementation delivers the outdated-version warning banner as a theme
override, as the PR doc and Phase 6 plan describe. `tools/overrides/main.html`
extends `base.html` and fills the `outdated` block with "You're not viewing the
latest version." plus a link to latest; `convert.py` adds `OVERRIDES_DIR` and
`OVERRIDE_TEMPLATE`, sets `theme.custom_dir = "overrides"` on the merged config
(`setdefault("theme", {})`), and copies the committed template to
`zensical/overrides/main.html`; the regenerated `zensical.toml` carries
`custom_dir = "overrides"`. Both RED-review points are addressed: the
override-content test now pins a non-empty block (`<a ` and `latest`), and the
override lives in a top-level `overrides/` rather than under `docs/`. The change
verifies end to end at the build level: the full suite is 76 passed, `ruff` is
clean, `check_flatten` is clean against the baseline, and the built root page
carries the hidden `md-banner--warning` aside with the warning text, the
`<a href="../.">` latest link (`{{ '../' ~ base_url }}` with `base_url == "."`),
and the standard `outdated.html` unhide script. The #29 override does not leak
into the built site. Comment and commit hygiene conform: invariant-style
comments, no issue IDs in source, no emojis or em-dashes, UK spelling. All
findings below are Notes; none require a change to this issue.

## Findings

### Note: A pre-existing template publishes into the built site, outside #29's scope

The build publishes `site/documentation-assets/overrides/main.html`, an unused
raw template served as a static file. This is not the #29 override (which lives
in the top-level `overrides/` and correctly does not leak). It is the monorepo's
own `documentation-assets/overrides/main.html`, a `site_meta` override that is
copied wholesale as part of the `documentation-assets` assets tree into
`docs/documentation-assets/overrides/main.html` and thereby served. It predates
this change (present in the RED commit `b50d609`; the GREEN commit `9a38400`
touches no `documentation-assets` path) and was never wired via `custom_dir` in
the monorepo either, so it is inert in both builds, not a regression. It is
worth capturing because it is exactly the "stray static template in the built
site" class of leak that motivated placing the #29 override at top level: the
corpus already ships one such leak, and a future cleanup could drop or relocate
`documentation-assets/overrides/` (and, if the custom favicons that template
carried are wanted, wire them deliberately). No action for #29.

Location: `zensical/docs/documentation-assets/overrides/main.html` (source of
the served `site/documentation-assets/overrides/main.html`)

### Note: mkdir in copy_content relies on the wholesale rmtree, not exist_ok

`copy_content` creates the override directory with
`override_dir.mkdir(parents=True)` (no `exist_ok=True`). This is correct only
because the function `rmtree`s and rebuilds `output` at the top of every run, so
`output/overrides` never pre-exists. The invariant holds and is documented (the
output tree is owned wholesale), so this is safe as written; it is noted only as
a coupling to be aware of if the rmtree is ever made conditional.

Location: `tools/convert.py:338-340`

### Note: The show-on-non-latest behaviour is verified at build level only

The banner renders present-but-hidden in every built page
(`data-md-component="outdated" hidden`). The unhide is standard Material JS: the
`outdated.html` partial sets `el.hidden = false` when the bundle's version JS has
written `__outdated` to sessionStorage from `versions.json`. That end-to-end path
was proven in #28's two-version deploy and is not re-driven in a browser here,
consistent with the PR doc. No action; recorded so the manual staging check
(the issue's stated done-condition) is not assumed to be covered by the unit
suite.

Location: `zensical/overrides/main.html`, `docs/prs/29.md`

## Verification

- Tests: pass. `python -m pytest tools/ -q` gives 76 passed.
- Lint: `ruff check tools/` passes.
- Flatten: `python tools/check_flatten.py` exits 0 ("build clean against the
  baseline").
- Build (from `check_flatten`, `zensical/site`): `site/index.html` contains one
  `md-banner--warning` aside with "not viewing the latest version.",
  "Click here to go to latest", the `data-md-component="outdated"` hidden
  wrapper, and `<a href="../.">`. No `site/overrides/` directory: the #29
  override does not leak.
- Config: `zensical/zensical.toml` sets `custom_dir = "overrides"` under
  `[project.theme]`, and `[project.extra.version]` provides `provider = "mike"`,
  so `base.html`'s enclosing `config.extra.version` condition holds and the
  banner renders.
- Generated tree matches source: `zensical/overrides/main.html` is byte-identical
  to `tools/overrides/main.html`.
- Build artefacts (`zensical/site`, `zensical/.cache`) removed after inspection;
  git working tree clean.

## Recommendation

Approve. The implementation matches the design, passes the full suite, lint, and
the flatten gate, and the banner is confirmed in the built output without leaking
the #29 override. The three findings are Notes: a pre-existing stray-template
leak from `documentation-assets/` (out of scope, candidate for a later cleanup),
a benign mkdir coupling, and a reminder that the show-on-non-latest step is a
manual staging check rather than a unit-covered path. None block merge.
