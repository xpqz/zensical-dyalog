# Review: Issue #41 - documentation-assets must be a submodule, not copied files

Stage: tests-only (RED).

## Summary

The RED surface is two changed tests in `tools/test_convert.py` that replace the
old `test_copies_documentation_assets_into_docs` and
`test_does_not_carry_vcs_metadata_into_the_output`. Both new tests assert
observable filesystem behaviour, are named as behavioural sentences, and fail
for the stated reasons. `python -m pytest tools/test_convert.py -q` gives 2
failed, 37 passed, matching the expected RED, and the failure messages are the
ones documented in `docs/prs/41.md`. `ruff check tools/` is clean. The two new
tests, taken with the unchanged stray-removal tests, form a sound two-sided
constraint on the reworked wipe. No blocking issues. The findings below are
forward-looking notes for GREEN, not defects in the test surface.

## Findings

### Note: Tests fail for the correct reasons and match the expected RED

`test_does_not_copy_documentation_assets` fails with `AssertionError: assert not
True`: convert still runs the `copytree(source/documentation-assets, ...)` at
`tools/convert.py:317`, so `docs/documentation-assets` exists.
`test_preserves_the_documentation_assets_submodule_on_regen` fails with
`AssertionError: assert 'body {}\n' == 'SUBMODULE SENTINEL\n'`: the wholesale
`shutil.rmtree(output)` at `tools/convert.py:313` deletes the planted sentinel
and the re-copy restores the source's `body {}\n`. Both failures are genuine and
target the two behaviours GREEN must change (stop copying; preserve the checkout
across the wipe).

Location: tools/test_convert.py:60, tools/test_convert.py:68

### Note: The new tests plus the unchanged stray-removal test are a good two-sided constraint

`test_rerun_removes_stray_files_planted_in_the_output` requires the re-run to
delete `junk.txt` and `docs/stray.md`, while
`test_preserves_the_documentation_assets_submodule_on_regen` requires it to keep
`docs/documentation-assets/`. Together these forbid both a wholesale wipe and a
"never touch docs/" no-op, so GREEN must clean convert-owned paths while sparing
the submodule path. The planted sentinel is a plain directory with no gitlink,
so preservation must key on the path, not on submodule detection, which matches
the design in `docs/prs/41.md`. `test_two_runs_produce_byte_identical_output`
and `test_never_writes_to_the_source_tree` remain consistent under the new
behaviour.

Location: tools/test_convert.py:68, tools/test_convert.py:136

### Note: Fixture still carries the documentation-assets .git gitlink; cleanup is correctly deferred to GREEN

`tools/fixture_tree.py:161` still plants `documentation-assets/.git`, and the
VCS-metadata test it supported was replaced rather than kept. This is correct for
a tests-first stage: the RED diff touches only tests. `docs/prs/41.md` schedules
removing the dead `.git` exclusion (`tools/convert.py:310,316-320`) and the
associated fixture entry in GREEN, since a real submodule legitimately carries a
`.git`. The remaining `documentation-assets` references in the fixture config
(extra_css, logo, favicon) stay valid: the submodule provides those files and the
config tests (for example `test_takes_extra_css_from_the_root_alone`) still expect
them.

Location: tools/fixture_tree.py:159, tools/convert.py:310

### Note: convert.py docstrings must be updated in GREEN to match the new invariant

The module docstring (`tools/convert.py:9`) and `copy_content` docstring
(`tools/convert.py:292-306`) still state that convert copies the
documentation-assets tree and excludes its `.git`. No test asserts docstring
content, so this does not affect the RED result, but GREEN must rewrite these so
the stated invariants match the code that stops copying and preserves the
checkout.

Location: tools/convert.py:9, tools/convert.py:292

## Verification

- Tests: fail as expected. `python -m pytest tools/test_convert.py -q`: 2 failed,
  37 passed. The two failures are the new tests, with the documented messages. No
  import or collection errors.
- Lint: `ruff check tools/` passes.
- APL semantics: not applicable. These are Python filesystem tests for the
  conversion tooling; they contain no APL expressions.

## Recommendation

Approve. The test surface correctly and minimally pins the target behaviour,
fails for the right reasons, and matches the expected RED. Proceed to GREEN.

# Review: Issue #41 - documentation-assets must be a submodule, not copied files

Stage: implementation (GREEN). Commit af5aca3 on top of RED commit 8f0a581.

## Summary

The implementation delivers exactly what the issue and PR notes describe:
`copy_content` no longer copies `documentation-assets`, the wholesale
`rmtree(output)` is replaced by `_clean_owned`, which regenerates convert-owned
paths while sparing `docs/documentation-assets`, and the root `docs` copy now
merges with `dirs_exist_ok=True` because `docs/` persists to hold the submodule.
The 93 copied files are replaced by a submodule gitlink at
`zensical/docs/documentation-assets` pinned to `c6ad6b7`, with a matching
`.gitmodules`. The dead `.git` ignore and the fixture gitlink are gone, and both
docstrings state the new invariant. All verification passes: 76 tests, ruff
clean, `check_flatten` clean, and a live `convert.py` re-run leaves the submodule
at `c6ad6b7` with a clean working tree and no git churn (deterministic
regeneration). `_clean_owned` is symlink-safe and order-independent, and the
two-sided constraint (preserve the submodule, remove strays and orphans) holds.
One Minor robustness gap: the post-copy markdown walk descends into the
submodule. It does not currently dirty it, but the "leaves the submodule intact"
invariant is upheld by the submodule's content rather than by construction.

## Findings

### Minor: The H1-rewrite walk descends into the submodule

After copying, `copy_content` runs `for md_file in docs.rglob("*.md")` to rewrite
raw-HTML page titles. `docs/` now contains the preserved
`docs/documentation-assets` submodule, and that checkout carries a `README.md`,
so the walk reads a file inside the submodule on every run. `rewrite_h1` returns
the text unchanged because `README.md` has no raw `<h1`, so no write happens and
the checkout stays clean (verified: `git status` empty, submodule working tree
clean after a re-run). The invariant is therefore currently upheld, but by the
submodule's content, not structurally: any future `.md` added to
`documentation-assets` containing a standalone raw `<h1>...</h1>` line would be
rewritten in place, dirtying the submodule working tree and contradicting the
docstring's "leaving the submodule checkout intact". Recommend excluding the
preserved `documentation-assets` subtree from the rewrite walk (skip paths under
`docs/<ASSETS_DIR>`) so the ownership boundary that `_clean_owned` enforces for
the wipe is also enforced for the rewrite pass. Not merge-blocking: the tree is
an assets repo (CSS, fonts, images) and the current checkout has no such file.

Location: tools/convert.py:355 (the `rglob` loop); invariant claimed at
tools/convert.py:328-331.

### Note: _clean_owned is symlink-safe, deterministic, and preserves by name as designed

`_remove` unlinks symlinks (the `not path.is_symlink()` guard means a symlink to
a directory is unlinked, not followed into `rmtree`), so a stray symlink planted
in the output is removed without touching its target. Removal is independent per
entry, so `iterdir()` order does not affect the result. Preservation keys on the
child name `documentation-assets` (`ASSETS_DIR`), which matches the design in
`docs/prs/41.md` and the RED note that the planted sentinel is a plain directory
with no gitlink, so preservation cannot rely on submodule detection. The
two-sided constraint is satisfied:
`test_preserves_the_documentation_assets_submodule_on_regen` keeps the submodule
path while `test_rerun_removes_stray_files_planted_in_the_output` and
`test_rerun_removes_output_files_whose_source_disappeared` remove strays and
orphans.

Location: tools/convert.py:295-315

### Note: Fixture retains a documentation-assets working file no test now reads

`fixture_tree.py:157-158` still creates `src/documentation-assets/css/main.css`.
Since convert no longer copies `documentation-assets` and no test asserts on a
copied `documentation-assets` file, this source-tree entry no longer feeds any
assertion. It is harmless (it mirrors the real source layout and lets
`test_does_not_copy_documentation_assets` exercise a realistic input), and the
`documentation-assets` references in the fixture `mkdocs.yml` remain load-bearing
for the config tests (`extra_css`, `logo`, `favicon`). No action required.

Location: tools/fixture_tree.py:157

### Note: Docstrings match the new behaviour; comment hygiene clean

The module docstring (tools/convert.py:9-11, 28-31) and the `copy_content`
docstring (tools/convert.py:327-331) now state the submodule invariant, and
`_clean_owned` carries a concise invariant docstring. The inline comment at
tools/convert.py:341 explains why `dirs_exist_ok` is needed (docs persists to
hold the submodule). No GitHub issue IDs, emojis, or em/en-dashes appear in the
source; the commit references the issue only in the `Closes #41` trailer.

## Verification

- Tests: pass. `python -m pytest tools/ -q`: 76 passed.
- Lint: `ruff check tools/` passes.
- Submodule preserved: after `python tools/convert.py`, the submodule HEAD stays
  `c6ad6b7`, its working tree is clean, and `git status` on the repo is empty
  (regeneration is deterministic and does not touch the submodule).
- End-to-end: `python tools/check_flatten.py` reports "OK: build clean against
  the baseline"; the built site carries `documentation-assets/css/main.css`
  (12062 bytes) resolved from the submodule. Generated `zensical/site` and
  `zensical/.cache` removed afterwards.
- Submodule wiring: `.gitmodules` declares the submodule at
  `zensical/docs/documentation-assets`; the index gitlink is pinned to
  `c6ad6b7`, matching the commit the source uses.
- APL semantics: not applicable (Python filesystem tooling, no APL expressions).

## Recommendation

Approve with minor changes. The change is correct, deterministic, and fully
verified: it swaps the copied tree for a pinned submodule, preserves it across
regeneration, and leaves no content churn. The single Minor finding (the
markdown-rewrite walk crossing into the submodule) is a latent robustness gap
rather than a current defect and can be hardened in a follow-up; it does not
block merge.
