# Review: Issue #28 - Wire zensical build into the versioned deploy (tests-only RED)

## Summary

Tests-only RED review. The committable code change for this issue is a single
addition to the conversion script: emit `site_url` in the generated
`zensical.toml`, because Zensical only version-prefixes `site_url`
(`docs.dyalog.com/21.0/`) when it is set, and the source config sets none. The
RED surface is two tests added to `tools/test_convert.py`, pinning the value to
the production canonical `https://docs.dyalog.com/` at two boundaries: the
merged config and the serialised toml. Both fail with `KeyError: 'site_url'`
exactly as intended; the other 71 tests pass and ruff is clean. The value is
correctly grounded: `tools/jenkins/RELEASE.md` in the source repo names
`docs.dyalog.com` as production and `dyalog.github.io/documentation/<ver>/` as
staging, and the source root `mkdocs.yml` has `site_name` but no `site_url`. No
defects found. Recommend approval to proceed to GREEN.

## Findings

No defects. The two tests are well-named (each reads as a sentence describing
the behaviour), assert observable output rather than implementation, and follow
the file's established two-layer convention (a `merged`-fixture test for config
shape, a `zensical.toml` test for the emitted artefact). The explanatory comment
states the invariant (why `site_url` is needed and why this value) without
narrative, issue IDs, emojis, or em-dashes. The changes are tightly scoped: 14
insertions, no source touched.

The observations below require no change.

### Note: The merged-config test pins the injection point to merge_configs

`test_sets_site_url_to_the_production_canonical` reads the `merged` fixture
(`convert.merge_configs(root, subs)`), so it requires `site_url` to be present
in `merge_configs`' output. This constrains GREEN to inject the constant at or
before `merge_configs`, not in `write_zensical_toml`. This is the right
boundary: it matches how every other config-shaping behaviour in the file is
tested (plugin dropping, extension folding, extra folding all assert against
`merged`) and keeps `write_zensical_toml` a pure serialiser, as its docstring
promises. Consistent, not a concern.

Location: tools/test_convert.py:268-272

### Note: Version-prefixing itself is not unit-verifiable, and correctly is not attempted

The pair pins the merged value and the serialised value; neither can verify that
Zensical actually rewrites the canonical to `https://docs.dyalog.com/21.0/` at
build time, since that needs a real build. That verification is correctly
deferred to the manual/CI step recorded in `docs/prs/28.md` (build with
`MIKE_DOCS_VERSION=21.0`, inspect a page's canonical). Appropriate scope for a
unit test.

Location: tools/test_convert.py:328-332

### Note: The trailing slash is load-bearing and is pinned exactly

Both tests assert the exact string `https://docs.dyalog.com/`, including the
trailing slash. This is the base form the fork's version-prefixing appends to
(yielding `/21.0/`), and matches the form proven in the local two-version
deploy. Pinning it by exact-match assertion is correct.

Location: tools/test_convert.py:272, tools/test_convert.py:332

## Verification

- Tests: fail as expected. `python -m pytest tools/ -q` gives 2 failed
  (`test_sets_site_url_to_the_production_canonical` and
  `test_zensical_toml_carries_site_url`, both `KeyError: 'site_url'`), 71 passed.
- Lint: `ruff check tools/` reports all checks passed.
- Scope: `git diff` shows only 14 insertions to `tools/test_convert.py`;
  `convert.py` and all other sources are unchanged.
- Grounding: `tools/jenkins/RELEASE.md` (source repo) confirms production is
  `docs.dyalog.com`; source root `mkdocs.yml` sets no `site_url`.

## Recommendation

Approve. The RED surface is minimal, correct, and correctly failing. Proceed to
GREEN: add the `SITE_URL = "https://docs.dyalog.com/"` constant and set it on the
merged config in `merge_configs`.

# Review: Issue #28 - Wire zensical build into the versioned deploy (implementation GREEN)

## Summary

Implementation review of GREEN commit `d32c131` on top of RED `889e877`. The
change greens the two RED tests at exactly the boundary the RED review
prescribed: `convert.py` gains a `SITE_URL = "https://docs.dyalog.com/"` module
constant and sets `merged["site_url"] = SITE_URL` in `merge_configs` (not in
`write_zensical_toml`, which stays a pure serialiser), with the `merge_configs`
docstring updated to record the new key. The regenerated `zensical.toml` differs
from the RED tree by exactly one line (the `site_url` entry); nothing else in the
generated tree moved. Two new artefacts support the deploy:
`tools/requirements-deploy.txt` pins the `squidfunk/mike` fork by commit, and
`docs/deploy.md` is the manual runbook. The full suite is 73 passed, ruff is
clean, `check_flatten` (a real Zensical build) exits 0, and I verified the
load-bearing behaviour end to end: a build with `MIKE_DOCS_VERSION=21.0` emits
`<link rel="canonical" href="https://docs.dyalog.com/21.0/">` on the root page
and the version-prefixed canonical on a deep page, with the page `__config`
carrying the `mike` version provider. One minor documentation defect in the new
runbook (an install path that does not resolve from this repo) and two notes; the
code change itself is correct, minimal, and in scope.

## Findings

### Minor: deploy.md install step references a file absent from this repo

The install block's first line is `pip install -r tools/requirements-docs.txt`
with the inline comment "(from the source monorepo)". This repo has no
`tools/requirements-docs.txt`; the file lives in the sibling source monorepo at
`../documentation/tools/requirements-docs.txt`. The other two install lines
(`pip install -e tools/` and `pip install -r tools/requirements-deploy.txt`)
correctly reference this repo's `tools/`, and the surrounding runbook has the
reader operating from this repo (it later runs `cd zensical/`). A reader
following the runbook verbatim from this repo hits a "file not found" on the
first step. Point the path at the source checkout (for example
`../documentation/tools/requirements-docs.txt`) or state the working directory
explicitly.

Location: docs/deploy.md:25

### Note: The SITE_URL comment asserts an override mechanism that does not yet exist

The constant's comment ends "Staging/test builds override this at deploy time."
No override path exists in the code or the runbook: the per-environment
`site_url` override is deferred to #30, and `docs/deploy.md` only says a
different-host build "should override" it. The sentence reads as a statement of
current behaviour for a mechanism that is not wired up. Reword to the constraint
it is really expressing (a build served from another host needs its own
canonical) rather than asserting an override happens. Not blocking; the rest of
the comment is a clean invariant with no issue IDs, emojis, or em-dashes.

Location: tools/convert.py:89-90

### Note: Staging builds emit a production canonical by design

Because `site_url` is hardcoded to the production host, a staging deploy at
`dyalog.github.io/documentation/<ver>/` emits
`<link rel="canonical" href="https://docs.dyalog.com/<ver>/">`. This is
intentional and documented (`docs/deploy.md` and the PR doc): internal links are
relative so navigation still resolves under the staging prefix, and pointing the
canonical at production is the correct SEO posture. The issue's done-criterion
(staging `/21.0/` URLs resolve) is unaffected by `site_url`. Flagged only so the
deferred #30 per-environment override stays a conscious follow-up.

Location: tools/convert.py:90, docs/deploy.md:50-53

## Verification

- Tests: pass. `python -m pytest tools/ -q` gives 73 passed. The two former RED
  tests (`test_sets_site_url_to_the_production_canonical`,
  `test_zensical_toml_carries_site_url`) now pass.
- Lint: `ruff check tools/` reports all checks passed.
- Generated config: `grep '^site_url' zensical/zensical.toml` yields
  `https://docs.dyalog.com/`; `git diff 889e877 d32c131 -- zensical/` is a single
  insertion.
- check_flatten: exit 0 ("build clean against the baseline"). This is a real
  Zensical build, so the new config is exercised.
- Version-prefixing (the load-bearing claim): built from `zensical/` with
  `MIKE_DOCS_VERSION=21.0`. Root `site/index.html` canonical is
  `https://docs.dyalog.com/21.0/`; deep page
  `site/object-reference/objects/button/index.html` canonical is
  `https://docs.dyalog.com/21.0/object-reference/objects/button/`; page `__config`
  provider is `mike`. Build artefacts (`site/`, `.cache/`) removed afterwards.
- Injection point: `merged["site_url"]` is set in `merge_configs`;
  `write_zensical_toml` is unchanged, matching the RED review's prescription.
- Comment hygiene: no GitHub issue IDs in source comments; references live in the
  commit, PR doc, and this review. No emojis or em-dashes in the changed source or
  docs.
- Scope: CI, GitHub Pages, and the per-environment `site_url` override are absent
  by instruction (deferred to #30); their absence is not faulted.

## Recommendation

Approve with minor changes. The code change is correct, minimal, tested, and the
version-prefixing behaviour is verified end to end. The one minor item
(`docs/deploy.md` install path) is documentation-only and does not block the
mechanism; fixing it, and optionally rewording the `SITE_URL` comment, would tidy
the runbook before #30 builds on it.
