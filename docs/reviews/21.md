# Review: Issue #21 - Generate the flattened project and validate under Zensical

Stage: tests-only (RED), Cycle 1. The branch delivers #21 and #22 together
under the scope change recorded in docs/prs/21.md.

## Summary

The reshaped surface is coherent and the RED run reproduces exactly (9
failed, 27 passed: 7 stubs, 2 behavioural changes to convert). The
committed baseline is empirically exact: an independent parse of a fresh
full-corpus build yields precisely the 28 entries in known-issues.txt,
zero differences either way. Three things must change before the tests
are committed. First, the provenance story attached to the baseline is
wrong: the site-root-absolute link from issue 15 is not in the baseline,
is not reported by the checker at all, and is not fixed upstream. Second,
the gate as specified can pass silently when it should fail: nothing
pins the parsed issue count to the checker's own summary line, a crashed
build yields an empty issue set that is a subset of any baseline, and the
parser matches only Warning severities. Third, the scope change that
retires the MkDocs oracle and legacy checkers exists only in the PR doc;
issues 21 and 22 still state the old contract they will be closed
against.

## Findings

### Major: the baseline provenance misstates what the checker sees and what is fixed

The known-issues.txt header and the PR doc both describe the 28 entries
as 8 caption anchors, 1 site-root-absolute link "fixed upstream" and 19
others. Measured against a fresh build: the 28 are 8 caption anchors plus
20 pre-existing broken links and anchors, and no absolute-link entry
exists in either the baseline or the checker output. Zensical leaves
absolute URLs unchanged and does not validate them, so the checker
cannot report that link. Nothing is fixed upstream: the link is still at
dotnet-framework-interface-guide/docs/installation.md:64 in the source
checkout and dyalog/documentation has no PR for it. Correct both
documents, and keep issue 15 open on its own track rather than treating
it as covered by this gate. When rewriting the header, drop the issue IDs
from it (state the caption restoration as "the caption numbering
extension in Phase 5" rather than by number).

Location: known-issues.txt:5, docs/prs/21.md (baseline paragraph)

### Major: the gate can pass silently when it should fail

Three failure modes of check_flatten currently end in an OK exit. A
future Zensical version that reshapes its diagnostic output (the tool is
pinned at 0.0.48; the format is not a contract) makes parse_issues return
the empty set, which is a subset of any baseline. A build that crashes
outright produces no diagnostics and run_build discards the return code.
A diagnostic emitted at a severity other than Warning is invisible to
the parser. The fix is cheap and belongs in the tested contract: parse
the checker's own "N issues found" summary line and fail when it is
missing or disagrees with the parsed set size, and fail on a non-zero
build exit. main's exit policy (new issue means exit 1, resolved-only
means exit 0, count mismatch means exit 1) is unit-testable with an
injected runner; the PR doc's decision to leave run_build integration-only
is fine, but the policy around it should not be.

Location: tools/check_flatten.py:35, tools/check_flatten.py:59, tools/test_check_flatten.py

### Major: the scope change is not recorded where it changes the contract

Issues 21 and 22 still require the MkDocs baseline build, the HTML diff,
the legacy link-checkers and the mkdocs.yml fallback; both will be closed
by this branch without any of that happening. The PR doc records the
decisions and the plan update is scheduled for Cycle 2, but the issues
are the statement of done and need a comment recording the reshaped
done-when (Zensical-only validation, single generated config, delivered
together) before the branch closes them.

Location: docs/prs/21.md (scope change section), issues 21 and 22

### Minor: the gate has no protection against new root-absolute links

The checker blindness that hides the issue-15 link also means nothing in
the pipeline flags a new root-absolute link introduced after cutover,
when this repository becomes the authoring home. The corpus audit on
issue 15 found exactly one such link, so today's residual risk is one
known defect; the forward risk is unbounded. A pure function that scans
the copied markdown for `](/` and `href="/` patterns, with its own
baseline semantics, would close the blind spot and fits the existing
test surface. Author's call whether it lands in this cycle or as a
follow-up issue.

Location: tools/check_flatten.py (surface)

### Minor: baseline entries that resolve should be pruned, and the policy stated

diff_issues treats current-is-a-subset-of-baseline as clean, which is
right for a gate, but a baseline entry that resolves and later regresses
becomes invisible (it is still in the baseline, so its reappearance is
not new). The tool already prints resolved entries; state in the
known-issues.txt header that resolved entries are removed when observed,
so the baseline only shrinks.

Location: known-issues.txt:1, tools/check_flatten.py:53

### Note: the baseline is empirically exact against the current tree

A fresh `zensical build` of the regenerated flattened tree reports
exactly the 28 issues in known-issues.txt (independent parse, ANSI
stripped, type-and-location keys; zero differences in either direction).
All 28 are Warning severity. The fixtures in test_check_flatten.py match
the real diagnostic shape, including the SGR escape forms.

### Note: the convert surface changes match the reshaped scope

The renamed tests pin caption removal from the merged config and
zensical.toml as the only emitted config; the oracle test and the yaml
import are gone; every other convert test is untouched. The RED
decomposition (7 NotImplementedError in check_flatten, 2 behavioural in
convert) reproduces exactly as the PR doc records.

## Verification

- Tests: fail (expected RED). 36 tests: 9 failed, 27 passed, matching
  the PR doc. Run in the local venv (Python 3.11, pytest 9.1.1).
- Integration evidence gathered for the review: full-corpus
  `zensical build` of the current tree completes in 9.5s (warm) with 28
  issues; independent parse matches known-issues.txt exactly.
- APL semantics step: not applicable, no APL expressions.

## Recommendation

Request changes. Scope: correct the baseline and PR-doc provenance (the
absolute link is neither in the baseline nor fixed upstream), add the
summary-count and build-exit failure modes to the tested contract, and
record the scope change on issues 21 and 22. The two Minors are the
author's call. The baseline data itself and the reshaped convert surface
are sound.

### Minor: merge_configs docstring contradicts the code on caption

The docstring says "The monorepo and site-urls plugins are removed;
caption is retained in the merged dict (serialisers decide its fate)."
The code removes caption too: DROPPED_PLUGINS is
frozenset({"monorepo", "site-urls", "caption"}) and the plugin filter at
the end of the function drops all three. The "serialisers decide its
fate" clause is a leftover from the two-serialiser design (the retired
write_mkdocs_yml kept caption, write_zensical_toml dropped it); there is
now one serialiser and it does not touch plugins. The net output is
correct (the emitted zensical.toml plugins are exactly privacy, search,
macros, minify), so this is a documentation defect, not a behaviour bug,
but it is the kind of stale narrative the project comment rules forbid
and it flatly misstates what the function does. The same docstring's
"extra_javascript keeps one javascripts/mathjax.js entry plus external
URLs" describes an action the function does not perform (extra_javascript
is inherited from the root config unchanged; no consolidation happens
here). Rewrite the two sentences to state that monorepo, site-urls and
caption are all dropped, and drop the extra_javascript claim or restate
it as inherited-from-root.

Location: tools/convert.py:143-145

### Note: the count guard fails safe on any key collision

assess compares the checker's summary count against len(parse_issues(...)),
a set of normalised keys. If Zensical ever emitted two diagnostics that
normalise to the same "<type> @ <file>:<line>:<col>" key, the set would
collapse them and the count guard would trip even on a legitimate build.
The current corpus has no such collision (summary 28 equals 28 parsed
keys), and the guard errs toward failing, which is the safe direction for
a gate, so this is an observation rather than a defect. The module comment
already names the count guard as the backstop for parser fragility, which
is the right framing.
